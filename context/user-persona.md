## 画像全景

```
内部用户
  ├── P1 产品线接入开发者   → BFF / SDK / 软件包使用者
  └── P2 平台运营人员       → 运营后台，跨租户监控与干预

外部用户（企业侧）
  ├── P3 企业集成开发者     → OpenAPI / Webhook 接入
  ├── P4 流程管理者         → 企业管理员，全局追踪与审计
  └── P5 流程发起者         → 业务人员，日常发起签署流程

外部用户（终端侧）
  └── P6 任务执行者         → 签署人 / 填写人 / 用印审批人 / 抄送人
```

## P1：产品线接入开发者（内部）

**代表角色**：国内站/国际站/天印产品线的研发人员

**一句话描述**：负责将 ePaaS 底座能力适配到各自产品线，是 BFF 层、内部 SDK 和软件包的主要消费者。

### 背景
- 来自各产品线研发团队（国内站、esignGlobal、天印 V6.0），非平台侧人员
- 通过 BFF 层、内部 SDK、软件包与底座交互，不直接暴露给外部客户
- 需要在底座统一能力基���上，完成本产品线的差异化定制（页面模块、流程参数、规格控制）
- 天印方向的开发者还需兼顾私有部署（混合云）环境的适配

### 核心目标
1. 快速理解底座提供了哪些能力、扩展点在哪里
2. 通过配置参数/BFF 封装完成差异化，不需要改底层代码
3. 底座升级时影响范围可预判，不被动挨打

### 主要痛点
- 底座内部规范文档滞后，靠口耳相传或读源码理解
- 扩展点设计不足，被迫绕过标准路径实现需求（后患大）
- 底座变更缺少提前通知，联调成本高
- 混合云环境与 SaaS 环境行为差异没有系统性说明

### 使用场景
- 新功能接入：了解底座新能力 → 评估扩展方案 → 通过 BFF / 软件包实现 → 联调
- 日常维护：处理产品线特有问题，与平台侧对齐边界
- 版本适配：底座升级时评估影响、回归测试

### 对产品设计的影响
- 新能力设计需明确扩展点和配置参数，不能只有硬编码路径
- 内部技术规范文档需与代码同步更新
- 底座 API 变更需提前告知并说明影响范围

## P2：平台运营人员（内部）

**代表角色**：ePaaS 平台运营团队、客户成功人员

**一句话描述**：通过运营后台对全平台流程进行跨租户监控、异常处置和数据分析，是平台健康运转的保障者。

### 背景
- 属于平台内部团队，拥有超级管理员权限
- 视角跨租户、跨产品线，可查看和干预任意企业的信封流程
- 日常工作包括：客户投诉处理、异常流程排查、运营数据统计
- 部分操作（如强制撤销、数据修正）需要走审批流

### 核心目标
1. 快速定位客户反馈的问题（找到对应信封、还原操作链路）
2. 对平台级异常（大批量超时、系统故障导致的中断）能批量处置
3. 提供准确的运营数据支撑业务决策

### 主要痛点
- 问题排查路径长：客户描述模糊 → 找信封 → 查日志 → 定位原因，每步都靠手动
- 批量处置能力弱，大面积异常时需逐条操作
- 运营数据分散，无法快速出报表
- 高权限操作缺乏操作留痕和审批机制，合规风险高

### 使用场景
- 客诉处理：根据客户提供的信息查找信封 → 查看完整操作日志 → 判断责任方 → 处置或给出结论
- 异常巡检：定时查看超时/中断/异常状态的信封，批量催签或转交
- 数据统计：按产品线、租户、时间段统计签署量、完成率、异常率

### 对产品设计的影响
- 运营后台需要强搜索能力（跨租户、多维度筛选）
- 批量操作是核心需求，单条操作效率无法满足实际工作量
- 高权限操作必须有操作日志和审批链路

## P3：企业集成开发者（外部）

**代表角色**：客户企业的技术开发者、系统集成工程师

**一句话描述**：负责将签署 ePaaS 的 OpenAPI 能力接入企业自有系统，追求接入稳定、文档清晰、问题可自助排查。

### 背景
- 来自客户企业技术团队，非 ePaaS 平台内部人员
- 通过公开 OpenAPI、Webhook 与平台交互，接触不到底层实现
- 对签署业务本身了解有限，主要关注接口的输入输出和异常处理
- 可能同时对接多套系统，ePaaS 只是其中一个供应商

### 核心目标
1. 快速完成接入，减少联调来回的沟通成本
2. 出问题时能自助排查，不依赖平台支持
3. 接口稳定，升级时有足够提前通知

### 主要痛点
- 公开文档与实际接口行为不一致，靠猜或靠问平台支持
- Webhook 回调丢失、乱序，且没有自助重推工具
- 错误码语义模糊，排查路径长
- ��环境（SaaS / 私有部署）差异未在文档中说明

### 使用场景
- 首次接入：阅读文档 → 申请 AppId → 沙箱调试 → 生产上线
- 日常运维：监控回调状态、处理偶发异常
- 版本升级：评估接口变更影响、回归测试

### 对产品设计的影响
- 公开接口需严格版本管理，Breaking Change 需提前通知
- Webhook 需提供投递日志和手动重推能力
- 错误码需有明确语义和处理建议文档

## P4：流程管理者（外部 · 企业侧）

**代表角色**：企业管理员、合规专员、IT 负责人

**一句话描述**：被企业授权后，负责监控本企业所有签署流程进度、处理异常，并在合规场景下提供完整操作记录。

### 背景
- 拥有企业级管理员权限，可查看本企业所有信封（非跨租户）
- 不参与具体签署动作，关注"全局视角"
- 合规场景下需要提供流程记录作为法律或审计证据
- 在大型企业中可能有部门/业务线维度的权限分级需求

### 核心目标
1. 随时掌握所有流程进度，发现卡点及时介入
2. 对异常信封（超时、中断、撤销）能快速处置
3. 能提供满足合规要求的完整操作记录

### 主要痛点
- 流程数量多，缺乏有效预警机制，靠人工巡检效率极低
- 异常信封需逐条处理，批量操作能力弱
- 审计日志导出格式不符合内部合规模板
- 权限粒度太粗，无法按部门分配管理范围

### 使用场景
- 日常巡检：查看待处理 / 超时 / 异常信封列表 → 催签或转交
- 异常处置：撤销错误流程、延期过期信封
- 审计导出：按时间段、业务类型导出签署操作记录

### 对产品设计的影响
- 列表页需要多维筛选 + 状态预警（超时即将到期等）
- 批量催签、批量转交是高频刚需
- 操作日志需完整可导出，支持自定义字段和格式

## P5：流程发起者（外部 · 企业侧）

**代表角色**：企业业务人员（销售、HR、采购、法务等）

**一句话描述**：在日常工作中需要发起签署流程，追求操作简单、流程合规、进度可追踪。

### 背景
- 非技术背景，通过平台前端页面操作
- 签署是工作流程中的一个环节，不是核心工作，不愿意花时间学习
- 对签署合规性有要求（合同、协议类场景尤其如此）
- 发起频率因角色差异较大（高频：HR 批量入职；低频：采购偶发合同）

### 核心目标
1. 快速发起一份正确的签署流程，不出错
2. 流程发起后能知道进度，不需要反复追人
3. 完成后方便取到签署完成的文件

### 主要痛点
- 模板配置复杂，首次发起容易出错（签件位、参与顺序配置错误）
- 发起后进度不透明，需要手动催促参与者
- 错误发起后撤销再重来成本高，且撤销后参与者也有感知
- 不同业务场景的签署结构差异大，模板复用率低

### 使用场景
- 新建信封：选模板 → 填写参与者信息 → 上传文档 → 配置签件位 → 发起
- 进度追踪：查看信封状态、对超时任务催签
- 文件归档：流程完成后下载签署文件

### 对产品设计的影响
- 发起流程需引导式设计，降低配置出错率（尤其是参与顺序和签件位）
- 进度通知（站内 + 消息推送）是刚需，不能只靠主动查询
- 模板需支持按业务场景分类，支持常用配置复用

## P6：任务执行者（外部 · 终端侧）

**代表角色**：签署人、填写参与人、用印审批人、抄送人

**一句话描述**：被指派完成信封中某个具体任务的参与者，目标是低摩擦地完成自己的那一步，然后离开。

### 背景
- 可能是企业内部员工，也可能是外部合作方（客户、供应商）
- 通常通过短信 / 邮件收到链接，点击后进入完成操作，完成即离开
- 对签署平台本身没有归属感，更关注"我能不能快速完成"
- 移动端使用频率高，设备和网络环境多样

### 子角色差异

| 子角色 | 核心任务 | 特殊关注点 |
|--------|----------|------------|
| 签署人 | 阅读文件，在指定位置完成电子签名 | 签名的法律效力说明、实名认证流程 |
| 填写参与人 | 在文件中填写指定字段（姓名、金额、日期等） | 字段校验规则是否清晰、必填提示 |
| 用印审批人 | 审核文件内容，决定是否加盖电子印章 | 审核拒绝时的意见填写和流程说明 |
| 抄送人 | 接收流程通知和最终文件，无需操作 | 文件可访问性和通知及时性 |

### 核心目标
1. 清楚知道"我需要做什么"，无需学习成本
2. 快速完成操作，流程顺畅不卡壳
3. 明确确认自己操作已成功提交

### 主要痛点
- 链接失效或 Ticket 过期，找不到入口，又不知道找谁重发
- 不清楚自己需要签哪里、填什么，需要联系发起人
- 移动端阅读文件体验差（缩放不便、签名区域过小）
- 操作完成后反馈不明确，不确定是否已成功提交

### 使用场景
- 标准路径：收到通知 → 点击链接 → 验证身份 → 阅读文件 → 完成任务 → 收到完成确认
- 异常场景：链接过期后重新获取入口、拒签并填写原因

### 对产品设计的影响
- 签署入口页需清晰说明当前任务类型和所需操作
- Ticket 失效后的重新获取路径必须顺畅，不能依赖联系发起人
- 移动端是主力场景，文件阅读和签名交互需专门优化
- 操作成功页面和通知是必须项，不能省略
