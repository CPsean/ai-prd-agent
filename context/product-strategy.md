# 产品策略：签署 ePaaS

> AI 上下文文件。写 PRD 或做方案设计时，用此文件判断优先级、边界约束和正确决策方向。

---

## 迭代策略原则

### 原则 1：防御性架构优先（Architecture For Hybrid）

**规则**：任何底层新增能力设计，必须通过"天印 V6.0（混合云）兼容性审查"。底层 API 与数据模型严禁写死上层业务逻辑。

**实操含义**：
- 新功能设计时，必须考虑私有部署（混合云）场景能否支持
- 不能把 SaaS 专属依赖（如云存储特定实现、外部 SaaS 服务）强耦合进底层逻辑

---

### 原则 2：国内站迁移优先（Domestic Migration First）

**规则**：优先补齐国内站与旧系统主流程的功能对齐，降低迁移成本的底层核心能力（如整表工具、精度路由）享受最高优先级优化。

**实操含义**：
- 涉及国内站核心签署主流程的需求，优先级最高
- 迁移相关的兼容性需求不能以"技术债"为由推迟
- 评估工作量时，兼容旧系统的接入成本需纳入估算

---

### 原则 3：国际站增收驱动（International Revenue Driven）

**规则**：在业务应用层（BFF 与前端模块）的迭代中，资源向国际站直接关联客户付费意愿的功能倾斜（如角色认证、多语言高级模板等）。

**实操含义**：
- 国际站功能需求应明确标注"直接影响付费转化"还是"体验优化"
- 直接影响营收的功能 > 体验类优化

---

## 产品线优先级矩阵

| 需求类型 | 国内站 | 国际站 | 混合云（天印） |
|----------|--------|--------|----------------|
| 迁移兼容 | **P0** | — | — |
| 底层核心能力 | P0 | P1 | 兼容评审（必过） |
| 直接影响付费功能 | P1 | **P0** | — |
| 体验优化 | P2 | P1 | — |
| 混合云专属 | — | — | P2 |

---

## 当前不做的事（边界）

- 不对底层代码进行产品线专属定制（通过 BFF 层/配置参数解决差异）
- 天印 V6.0 无强制整体迁移要求（保持架构兼容即可）
- 底层 API 与数据模型不写死上层业务逻辑

---

## 写需求时的检查清单

在提交任何涉及底层能力的 PRD 前，确认：

- [ ] 该功能在混合云私有部署下是否可用或可降级？
- [ ] 是否影响国内站现有迁移主流程？
- [ ] 对国际站而言，是否与客户付费决策直接相关？
- [ ] 底层接口是否保持对三条产品线的一致性？
