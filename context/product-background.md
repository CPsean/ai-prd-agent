## 核心定位

**产品名称**：签署 ePaaS（电子签署平台即服务）

**核心定位**：全链路、高可配置的"开箱即用"电子签署 ePaaS 底座。

**架构主张**：大中小企业均可接入，轻/重均可部署。通过"统一底层架构 + 差异化上层封装"模式支撑业务。

**服务模式**：不仅提供底层接口，更直接向终端用户提供标准化的前端页面模块（发起页、表单管理页、签署页、预览页等）。

**业务赋能策略**：对于不同产品线，平台不直接定制底层代码，而是通过提供"软件化"、"BFF 层"、"配置参数"、"产品规格"等系统解耦解决方案填补差异，确保基础设施的绝对一致性与可扩展性。

## 三条产品线

### 1. 国内站（eSign 国内）

- **当前阶段**：核心营收来源，目前由旧系统支持
- **首要目标**：**完整且平稳迁移** —— 将旧系统无缝迁移至新 ePaaS 底座
- **优先级**：最高，影响主营收盘

### 2. 国际站（esignGlobal）

- **当前阶段**：底层平台集成已完成
- **首要目标**：**功能补齐与商业变现** —— 聚焦海外增长渠道需求，完善高价值功能以驱动营收
- **优先级**：次之，直接关联客户付费意愿的功能（如角色认证、多语言高级模板等）

### 3. 混合云产品（天印 V6.0）

- **当前阶段**：业务收缩期，无强制迁移要求
- **首要目标**：**架构兼容与余存** —— 新功能设计必须将混合云私有部署场景作为前置约束
- **优先级**：防御性，后续新功能设计必须考虑混合云私有部署场景

## 核心能力架构概览

底层核心引擎遵循领域驱动设计（DDD），由三大域构成：

### 1. 核心签署工作台域（Envelope Domain）

- **中枢调度与全生命周期管理**：以 `Envelope` 为核心聚合根，管理全生命周期，通过状态机驱动业务强一致性
- **任务流转与执行层**：由 `EnvelopeTask` 驱动，维护节点顺序、执行模式（并签/顺签），支持加签、转交、中断和批量转移
- **身份、角色与访问控制层**：抽象参与主体（`EnvelopeParticipant`），基于 `EnvelopeTicket`（JWT）与权限引擎支持跨租户和底层访问控制
- **文档与物料管理层**：由 `EnvelopeDocument` 统筹文档，支持阶段性版本（phase）记录与来源溯源
- **审计监控与策略组件层**：利用 `EnvelopeActivity` 沉淀状态机快照，采用可组装组件（如催签超时流），实现策略与模型解耦

### 2. 业务编排与流程模板域（Process Template Domain）

- 抽象高频流程参数（如角色、签件位符），减少重复填写成本
- 实现"管用分离"，通过强规则约束确保发起规范

### 3. 平台级基础设施域（Platform Infrastructure）

- **租户与账号层**（规格控制）
- **证书与印章层**（信任根与防伪）
- **开放网关与消息层**（API 鉴权与 Webhook 异步回调）

## 关键术语对照

| 术语 | 含义 |
|------|------|
| Envelope / 信封 | 一次完整的签署流程实例，核心聚合根 |
| EnvelopeTask / 任务 | 信封中的可执行任务节点（一个签署人的待办） |
| EnvelopeParticipant / 参与者 | 信封中的参与主体（发起人/签署人/抄送人） |
| EnvelopeDocument / 文档 | 信封中的业务文档 |
| EnvelopeTicket | 用于信封访问/签署的 JWT 票据 |
| BFF 层 | Backend For Frontend，对接不同产品线的适配层 |
| 天印 V6.0 | 混合云版本，需兼顾私有部署场景 |
| phase | 文档阶段性版本记录（ORIGIN → 各签署阶段） |
