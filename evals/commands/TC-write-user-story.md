# 测试用例：/write-user-story 命令

> 关联命令：`.claude/commands/write-user-story.md`
> 参考规范：`.claude/skills/user-story/SKILL.md`
> 前置条件：无特殊前置文件
> Codex 等效输入：「帮我写 [需求描述] 的用户故事」或「为 [功能] 生成 Gherkin 格式的开发用户故事」

---

## TC-WS-01 单一需求生成用户故事

| 字段 | 内容 |
|------|------|
| **状态** | — |
| **测试目标** | 验证单一需求输出格式正确、质量检查通过的用户故事 |
| **前置条件** | 无特殊前置条件 |

**测试输入**
```
/write-user-story 财务人员需要批量导出费用报销单为 Excel，当前只能逐条下载 PDF
```

**预期行为**
1. 生成 1 个用户故事
2. 格式严格遵循命令定义的结构
3. "As a" 使用具体角色（"财务人员"），不是泛用"用户"
4. "so that" 说明价值/动机，不是重复行动
5. 验收标准使用 Gherkin 格式（Given/When/Then）
6. When 只有一个，Then 只有一个

**检查要点**
- [ ] 输出包含"用户故事 [编号]"标题
- [ ] 包含"摘要"字段，以用户价值为中心
- [ ] "As a" 为"财务人员"（具体角色）
- [ ] "so that" 不是"可以批量导出"的行动重复，而是说明业务价值
- [ ] 验收标准有 Given/When/Then 格式
- [ ] 每个场景中 When 只有一个
- [ ] 每个场景中 Then 只有一个

## TC-WS-02 复杂需求自动拆分多个故事

| 字段 | 内容 |
|------|------|
| **状态** | — |
| **测试目标** | 验证包含多个独立动作或角色的需求被拆分为多个用户故事 |
| **前置条件** | 无特殊前置条件 |

**测试输入**
```
/write-user-story 管理员可以创建和编辑角色权限，普通用户可以查看自己的角色信息和已授权功能列表
```

**预期行为**
1. 识别到包含两个角色（管理员、普通用户）和多个独立动作
2. 拆分为至少 2 个独立用户故事
3. 每个故事独立成卡，有独立编号

**检查要点**
- [ ] 输出包含至少 2 个用户故事
- [ ] 管理员和普通用户的故事分开
- [ ] 每个故事的 "As a" 角色不同
- [ ] 每个故事有独立的验收标准

---

## TC-WS-03 质量检查项验证

| 字段 | 内容 |
|------|------|
| **状态** | — |
| **测试目标** | 验证输出前执行了质量检查，输出不含常见问题 |
| **前置条件** | 无特殊前置条件 |

**测试输入**
```
/write-user-story 用户可以在个人设置页修改昵称和头像
```

**预期行为**
1. 生成用户故事
2. 自检通过：角色具体、价值明确、When/Then 各一个
3. 如需拆分（修改昵称和修改头像可能是独立故事），则拆分

**检查要点**
- [ ] "As a" 不是泛用"用户"，而是具体角色（如"注册用户"、"会员"等）
- [ ] "so that" 说明动机（如"保持个人信息的准确性和个性化"），非重复行动
- [ ] 摘要以价值为中心，非功能描述（如"个性化个人形象"而非"修改头像功能"）
- [ ] 未出现多个 When 或多个 Then 的场景
