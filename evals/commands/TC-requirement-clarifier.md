# 测试用例：/requirement-clarifier 命令

> 关联命令：`.claude/commands/requirement-clarifier.md`
> 前置条件：无特殊前置文件
> Codex 等效输入：「帮我澄清这个需求：[需求描述]」或「我有个想法，帮我分析一下真实问题：[描述]」

---

## TC-RC-01 典型 X-Y 问题输入

| 字段 | 内容 |
|------|------|
| **状态** | — |
| **测试目标** | 验证 AI 能识别出用户提出的是解决方案而非真实问题，并追问 |

**测试输入**
```
/requirement-clarifier 我想给列表页加一个"导出 Excel"按钮
```

**预期行为**
1. 识别"加导出按钮"是解决方案（Y），追问背后的真实需求（X）
2. 提出 1-2 个追问，例如：谁需要导出？导出后做什么用？现在是怎么解决这个问题的？
3. **不**直接输出 RDD 卡片，等待用户回答

**检查要点**
- [ ] AI 未直接接受"加导出按钮"作为需求，而是质疑或追问
- [ ] 追问数量 ≤ 2 个（不过度追问）
- [ ] 追问聚焦在"谁/为什么/现在怎么做"而非技术实现

## TC-RC-02 需求已足够清晰时直接输出 RDD

| 字段 | 内容 |
|------|------|
| **状态** | — |
| **Codex 状态** | ✅ 2026-04-13 |
| **测试目标** | 验证清晰需求不过度追问，直接产出 RDD 卡片 |

**测试输入**
```
/requirement-clarifier 财务人员需要每月将费用报销单批量导出为 Excel，目前只能逐条下载 PDF，非常耗时，导致月末结账时大量等待。
```

**预期行为**
1. 识别需求已包含：角色（财务人员）、当前痛点（逐条下载）、业务场景（月末结账）
2. 直接输出 RDD 卡片，包含：核心洞察、JTBD、MVP 范围、成功指标
3. 如有必要，追问最多 1 个补充问题（非阻断性）

**检查要点**
- [ ] 输出包含 RDD 卡片结构（核心洞察、JTBD、MVP、成功指标）
- [ ] JTBD 格式：When [场景] / I want to [行动] / So I can [结果]
- [ ] 追问数量 ≤ 1 个
- [ ] 卡片中有具体可衡量的成功指标（如"月末导出操作时间从 X 分钟降至 Y 分钟"）

## TC-RC-03 需求极度模糊无法判断方向

| 字段 | 内容 |
|------|------|
| **状态** | ✅ 2026-04-08（修复后） |
| **测试目标** | 验证极度模糊输入时 AI 先说明假设再追问，不胡乱猜测 |

**测试输入**
```
/requirement-clarifier 优化一下用户体验
```

**预期行为**
1. 识别输入过于模糊，无法确定问题域
2. 说明"用户体验"范围过广，列出 2-3 个可能方向
3. 请用户确认方向后再继续
4. **不**直接输出 RDD 卡片

**检查要点**
- [ ] AI 明确指出输入太模糊，无法直接分析
- [ ] 提供了 2-3 个方向供用户选择（如：加载速度/操作流程/视觉反馈）
- [ ] 未输出 RDD 卡片
- [ ] 语气引导而非拒绝

## TC-RC-04 步骤6：完成 RDD 后自动保存并引导进入 /new-prd（串联流水线）

| 字段 | 内容 |
|------|------|
| **状态** | ✅ 2026-04-10 |
| **Codex 状态** | ✅ 2026-04-13 |
| **测试目标** | 验证 RDD 卡片生成后，AI 自动：提炼标题 → 创建 drafts/[标题]/rdd.md → 输出引导进入 /new-prd |
| **前置条件** | 无特殊前置条件 |

**测试输入**
```
/requirement-clarifier 财务人员需要每月将费用报销单批量导出为 Excel，目前只能逐条下载 PDF，非常耗时，导致月末结账时大量等待。
```

**预期行为**
1. 输出 RDD 卡片（含核心洞察、JTBD、MVP 方案、成功指标）
2. 从 RDD 内容提炼简洁的需求标题（2-6 字中文，如"报销单批量导出"）
3. 创建目录 `drafts/[提炼标题]/`
4. 在其中创建 `rdd.md`，内容为完整 RDD 卡片
5. 输出引导语，告知：
   - RDD 已保存至 `drafts/[标题]/rdd.md`
   - 下一步运行 `/new-prd [标题]`，AI 将自动读取 RDD，跳过需求方向讨论

**检查要点**
- [ ] `drafts/[提炼标题]/rdd.md` 文件存在
- [ ] rdd.md 内容包含完整 RDD 卡片结构（核心洞察 / JTBD / 验收标准 / MVP / 成功指标）
- [ ] 提炼的标题简洁（2-6 字），能概括核心功能，非原始输入的截断
- [ ] 引导语中明确包含 `/new-prd [标题]` 命令格式
- [ ] 引导语说明了"AI 将自动读取 RDD 内容，跳过需求方向讨论"

## TC-RC-08 RDD 输出格式完整性验证

| 字段 | 内容 |
|------|------|
| **状态** | — |
| **测试目标** | 验证 Step 5 输出的 RDD 卡片包含节 1（7 维表格）和节 2（6 小节）的完整结构 |
| **前置条件** | 无特殊前置条件 |

**测试输入**
```
/requirement-clarifier 销售人员需要在外出拜访客户时快速查看客户历史订单和跟进记录，目前只能回公司打开 PC 端 CRM 查看，错过商机。
```

**预期行为**
1. AI 完成 X-Y 诊断和 JTBD 追问（可能需要 1-2 轮交互）
2. 输出完整 RDD 卡片，包含：

   **节 1 需求摘要**（7 维表格）：
   - 业务对象
   - 用户角色
   - 使用场景
   - 核心动作
   - 预期价值
   - 成功指标
   - 需求优先级（P0-P3）

   **节 2 初步方案**（6 小节）：
   - 方案边界（做 / 不做 / 后续规划）
   - 功能模块（含新增/调整/复用标签）
   - 页面结构（树状列表）
   - 核心流程（主流程 + 异常流程）
   - 关键设计决策（A/B 方案对比 + 推荐理由）
   - 风险提示（含编号 P+数字）

**检查要点**
- [ ] RDD 包含"节 1 需求摘要"，为表格格式，含 7 个维度
- [ ] RDD 包含"节 2 初步方案"，含 6 个子节
- [ ] 方案边界明确区分了"本版做"/"本版不做"/"后续规划"
- [ ] 关键设计决策有 A/B 对比（不止一种方案）
- [ ] 风险提示有编号（P1/P2 等）
- [ ] 总字数节 2 部分 ≤ 400 字（聚焦产品决策）
