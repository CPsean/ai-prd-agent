# 测试用例：/update-prd 命令

> 关联命令：`.claude/commands/update-prd.md`
> 前置条件：`prds/` 目录下存在 v1.0 的 PRD

---

## TC-UP-01 小变更（版本号 +0.1）

| 字段 | 内容 |
|------|------|
| **状态** | — |
| **测试目标** | 验证小变更流程：归档旧版、更新日期、版本 +0.1、追加 CHANGELOG |
| **前置条件** | `prds/消息通知设置/prd.md` 版本为 v1.0 |

**测试输入**
```
/update-prd 消息通知设置

变更内容：根据评审意见，将 AC-1 中"误差不超过1次刷新周期"修改为具体时间"页面进入后2秒内（P95）完成数据同步"。
```

**预期行为**
1. 读取当前 `prd.md`（v1.0）
2. 将当前版本归档到 `archive/prd-v1.0.md`
3. 修改 `prd.md` 中 AC-1 内容
4. 更新 frontmatter：`version: "1.1"`、`updated: 今天日期`
5. 在 `CHANGELOG.md` 追加 v1.1 变更记录
6. 更新 `_registry.md` 中对应行的 `updated` 列
7. 输出质检报告

**检查要点**
- [ ] `archive/prd-v1.0.md` 存在且内容为旧版本
- [ ] `prd.md` 中 version 为 "1.1"
- [ ] `prd.md` 中 updated 为今天日期
- [ ] `CHANGELOG.md` 中存在 "v1.1" 章节，记录了本次变更
- [ ] AC-1 内容已更新，不再包含"刷新周期"模糊描述

---

## TC-UP-02 重大重构（版本号升主版本）

| 字段 | 内容 |
|------|------|
| **状态** | — |
| **测试目标** | 验证重大变更时版本号正确升级（1.x → 2.0） |
| **前置条件** | 任意 PRD 存在（版本为 1.x） |

**测试输入**
```
/update-prd 消息通知设置

变更类型：重大重构
变更内容：重新梳理整个通知体系，增加"通知分组"功能，将原有 3 种通知类型扩展为 8 种，新增渠道维度（App推送/邮件/短信），页面结构全部重写。
```

**预期行为**
1. AI 判断本次变更为重大重构（涉及结构性调整）
2. 将当前版本归档（如 `prd-v1.1.md`）
3. 更新版本号为 `2.0`（而非 1.2）
4. CHANGELOG 中说明版本升级原因

**检查要点**
- [ ] `prd.md` 中 version 为 "2.0"（非 "1.2" 或其他小版本）
- [ ] 旧版本已归档
- [ ] CHANGELOG 中有"重大重构"相关说明
- [ ] 如果 AI 不确定是否应升主版本，应先询问用户确认

---

## TC-UP-03 变更后质检自动触发

| 字段 | 内容 |
|------|------|
| **状态** | — |
| **测试目标** | 验证 /update-prd 执行后自动输出质检报告 |
| **前置条件** | 任意 PRD 存在 |

**测试输入**
```
/update-prd 消息通知设置

变更内容：删除边界条件章节中的所有内容（模拟破坏文档质量）。
```

**预期行为**
1. 执行变更（删除边界条件内容）
2. 变更完成后自动运行质检
3. 质检报告应检测到 G4-01（边界条件章节缺失）为 ❌

**检查要点**
- [ ] 质检报告在变更描述后自动输出，无需用户额外触发
- [ ] 质检报告中 G4-01 标记为 ❌ 未通过
- [ ] AI 提示该项为阻断项，建议修复后再推进 PRD 状态
