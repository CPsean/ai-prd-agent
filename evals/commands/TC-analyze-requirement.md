# 测试用例：/analyze-requirement 命令

> 关联命令：`.claude/commands/analyze-requirement.md`
> 参数格式：`/analyze-requirement [需求描述]`
> 前置条件：无特殊前置文件
> Codex 等效输入：「深度分析一下这个需求：[描述]」

---

## TC-AR-01 正常需求分析（信息充足）

| 字段 | 内容 |
|------|------|
| **状态** | — |
| **Codex 状态** | — |
| **测试目标** | 验证完整需求描述下，分析报告包含所有必要章节且存至 analysis/ |
| **前置条件** | `analysis/` 目录存在 |

**测试输入**
```
/analyze-requirement 企业客户希望为员工批量开通账号，支持 CSV 导入，导入后自动发送激活邮件。当前只能逐个手动创建，50人以上团队入驻时运营成本极高。
```

**预期行为**
1. 识别需求来源方（运营/企业客户）、核心痛点（手动逐个创建）
2. 用户分析：定义至少 2 个使用场景
3. 优先级判断：MoSCoW 分类
4. 风险识别：技术风险（CSV 格式校验）、业务风险（重复账号处理）
5. 输出完整报告至 `analysis/YYYY-MM-DD-企业批量开户-analysis.md`
6. 提示可进入 Phase 2（`/design-solution`）

**检查要点**
- [ ] 报告文件存在于 `analysis/` 目录，文件名格式正确
- [ ] 包含背景、用户分析、功能范围、风险、分期规划五个章节
- [ ] MoSCoW 分类中至少有 Must 和 Should 两层
- [ ] 风险列表非空且有具体内容（非泛泛而谈）
- [ ] 提示了 Phase 2 的后续操作

## TC-AR-02 需求描述模糊时的处理

| 字段 | 内容 |
|------|------|
| **状态** | — |
| **Codex 状态** | — |
| **测试目标** | 验证模糊需求下 AI 列出待确认事项，不自行假设关键决策 |
| **前置条件** | 无特殊前置条件 |

**测试输入**
```
/analyze-requirement 做一个审批流程
```

**预期行为**
1. 识别出需求极度模糊（未说明审批对象、角色、流程）
2. 在"需求澄清"章节列出关键待确认事项
3. 仍输出分析报告框架，但在模糊处标注"待确认"
4. 不自行假设审批对象或业务流程

**检查要点**
- [ ] 报告中"需求澄清"章节有具体待确认问题（≥3 个）
- [ ] 待确认问题聚焦于业务决策（审批对象、角色、层级等），非技术实现
- [ ] 报告中模糊处有"待确认"标注
- [ ] 未自行假设审批是请假审批还是费用审批等具体场景

## TC-AR-03 读取产品上下文文件

| 字段 | 内容 |
|------|------|
| **状态** | — |
| **Codex 状态** | — |
| **测试目标** | 验证分析时参考了 context/ 下的产品背景信息 |
| **前置条件** | `context/product-background.md` 和 `context/user-persona.md` 已填写 |

**测试输入**
```
/analyze-requirement 在现有工单系统中增加"工单转派"功能，主管可以将已分配的工单重新指派给其他客服
```

**预期行为**
1. 读取 `context/product-background.md` 了解产品架构
2. 读取 `context/user-persona.md` 获取用户角色定义
3. 分析中引用产品上下文（如现有角色体系、权限模型）
4. 用户分析章节中的角色与 persona 文件一致

**检查要点**
- [ ] 报告中引用了产品上下文信息（非泛泛的通用分析）
- [ ] 用户角色与 `user-persona.md` 中定义的一致
- [ ] 分析考虑了产品现有架构约束
