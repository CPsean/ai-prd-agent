# 数据飞轮规则

> 本文件定义 context 文件与 PRD 之间双向联动的完整规则。
> CLAUDE.md 持有核心触发表；本文件持有判断逻辑和边界定义。

---

## 飞轮两个方向

```
context/ ──────读入──────▶ PRD 输出质量提升
   ▲                            │
   └──────────写出──────────────┘
   （用户确认后写入，AI 不自动覆盖）
```

---

## 写出方向：触发条件与必须执行的动作

### 触发表

| 触发事件 | 涉及文件 | 必须执行的动作 | 跳过条件 |
|---------|---------|--------------|---------|
| `/requirement-clarifier` RDD 保存（Step 6.5） | `business-glossary.md` | 识别新术语 → 输出提议表格 → 等待用户确认 → 写入 | RDD 中无专有名词，或所有术语已在 glossary 中定义 |
| `/new-prd` PRD 移入正式区（Step 7） | `product-feature-map.md` | 提取新增功能节点 → 输出提议 → 等待用户确认 → 写入 | PRD 无新增功能节点（纯文档修订类） |
| `/new-prd` PRD 移入正式区（Step 7c） | `business-glossary.md` | 若 Step 6.5 术语追加未完成，此时补充提议 | 术语已在 clarifier 阶段完成追加 |
| `/update-prd` PRD 变更（Step 8） | `product-feature-map.md` | 新增功能节点时提议更新 | 本次变更仅修改已有功能的描述，无新节点 |
| `/update-prd` PRD 变更（Step 8） | `business-glossary.md` | 新增术语时提议追加 | 本次变更未引入新专有名词 |
| `/ingest-prd` 历史 PRD 录入（Step 9） | 两个文件 | 同步检查并提议缺失的术语和功能节点 | 已有完整 context 覆盖 |
| `/new-prd` PRD 移入正式区（Step 7 选 B 完成后） | `backlog/requirement-pool.md` | 扫描 §12 + 正文 TODO → 提议入池（BKL-FLY-001） | PRD 中无 TODO/OQ，或需求池不存在 |
| `/update-prd` PRD 变更完成后 | `backlog/requirement-pool.md` | 扫描新增 TODO/OQ → 提议入池（BKL-FLY-002） | 本次变更无新增 TODO/OQ，或需求池不存在 |
| `/requirement-clarifier` RDD 完成 / `/new-prd` 状态变化 | `backlog/requirement-pool.md` | 检查需求池中是否有关联条目 → 提议更新状态（BKL-FLY-003） | 需求池不存在或无匹配条目 |

### 触发规则：什么算"新术语"

满足以下全部条件时，才判定为需要追加的新术语：

1. 该词在 `context/business-glossary.md` 中**不存在**（精确匹配，忽略大小写和空格）
2. 该词是产品/业务专有名词，非通用技术词汇（如"接口"、"用户"不算）
3. 该词在 RDD 或 PRD 中出现 **≥ 2 次**，或出现在核心流程描述中

输出格式（Markdown 表格）：
```markdown
| 术语 | 建议定义 | 约束/备注 |
|------|---------|---------|
| 费用报销单 | 员工提交的费用报销申请记录，包含金额、类型、附件 | 须经财务主管审批 |
```

### 触发规则：什么算"新功能节点"

满足以下任一条件时，判定为需要更新 `product-feature-map.md`：

1. PRD §5 或 §7 中出现了 `product-feature-map.md` 中**未登记**的功能模块名称
2. PRD §7 中使用了新的功能编号前缀（`[AREA]` 部分之前未出现）
3. PRD 中明确说明"新增页面/新增入口/新增模块"

输出格式（两部分）：
```markdown
**建议追加到功能结构树（Mermaid）：**
[功能名] → [所属父模块]

**建议追加到前缀映射表：**
| 前缀 | 模块名 | 说明 |
|------|-------|------|
| EXP | 报销管理 | 员工费用报销相关功能 |
```

## 读入方向：执行 PRD 命令前的必读规则

| 写到哪个章节 | 必须读取（文件存在时） | 目的 |
|-------------|-------------------|------|
| PRD §4 需求对象与概念模型 | `context/business-glossary.md` | 避免重复定义已有术语；§4 只写本 PRD **新引入**术语 |
| PRD §5 功能结构 | `context/product-feature-map.md` | 只写新增/调整节点；完整结构引用此文件，不在 PRD 中全量复写 |
| PRD §7 功能清单 | `context/product-feature-map.md` | 获取已有模块的编号前缀，保证新功能编号不重复、格式一致 |

**读取时机**：按章节分步读取，非全量预读。只在写到对应章节时才读取，以减少不必要的 context 消耗。

## context 文件不存在时的处理

| 文件 | 首次触发时的行为 |
|------|---------------|
| `context/business-glossary.md` 不存在 | 询问用户是否以本次 RDD/PRD 内容创建初始术语表；用户确认后创建并写入 |
| `context/product-feature-map.md` 不存在 | 询问用户是否以本次 PRD 功能结构为基础创建初始功能树；用户确认后创建 |

## 强制规则（不可跳过）

1. **AI 不得自动写入 context 文件**——必须输出提议，等待用户明确确认（回复"确认"/"好的"/"写入"等）后才执行写入操作。
2. **提议不得为空**——触发条件满足时，AI 必须输出提议内容；不得以"本次无变化"跳过，除非能明确说明跳过原因。
3. **已有术语不重复提议**——写入前必须与现有 glossary 比对，不追加重复条目。
4. **已有功能节点不重复提议**——写入前必须与现有 feature-map 比对，不追加已登记的模块。

## 一致性检查（`/sync-docs` Step 5 扩展）

执行 `/sync-docs` 时，除检查 PRD/页面规格/原型的文档一致性外，还须检查：

| 检查项 | 判断标准 | 输出 |
|--------|---------|------|
| 术语孤岛 | PRD §4 中定义的术语是否已写入 `business-glossary.md` | 列出遗漏术语，提议补充 |
| 功能前缀未注册 | PRD §7 中使用的功能编号前缀是否已在 `product-feature-map.md` 登记 | 列出未注册前缀，提议补充 |
| glossary 与 PRD 用词一致性 | PRD 正文中的术语是否与 glossary 定义一致（无别名/变体用法） | 列出不一致用词，建议统一 |
