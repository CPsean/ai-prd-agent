# 跨企业印章授权权限与租户复制问题决策建议方案

一、 业务背景

随着 SaaS 平台深入劳务代理（如 FESCO）、集团管控（分子公司代章）等深水区业务，客户提出了“将本企业印章授权给另一企业直接使用”的需求。该需求要求在保持**发起方（代理方）**操作便捷性的同时，确保**授权方（资源方）**的法律主体地位与数据可审计性。

二、 现状定义

为了平衡效率与合规，本次评审方案的前提设定如下：

1.  **证书使用：** 采用“代章模式”，签署时底层调用**授权方（企业 C）**的数字证书（CA）。
    
2.  **意愿触发：** 采用“静默授权模式”，即企业 C 预先开启授权，企业 B 在有效期/限额内使用时无需 C 实时审批。
    
3.  **技术路径：** 采用“印章跨租户共享”，印章作为资源由 C 共享授权至 B 的资源库，由 B 在其租户空间可以针对任意文档发起用印流程。
    

三、 核心问题

1.  **签署主体判定：** 当 B 操作 C 的印章时，法律上的“签名人”是谁？
    
2.  **合同归属与可见性：** 合同默认存放在 B 侧，C 如何在不破坏 B 业务流的前提下实现合规审计与归档？
    
3.  **权限模型定义：** 在 PBAC 模型中，如何定义 C 的角色及其行为权限？
    
4.  **文档无法穿透，用印范围无法细化**：授权行为如果需要限制可盖章的范围如合同类型、是否存在代理条款等，是否可以支持？
    

四、 建议方案对比分析

针对 C 在签署流程中的角色定义，提出以下三种方案：

方案 1：作为“签署方”加入流程（显式会签）

将 C 作为一个自动签署的节点加入到 B 发起的签署流程中。

*   **优点：** 流程逻辑与现有签署模型完全一致，不需要改造底层模型。
    
*   **缺点：**
    
    *   用户体验差：外部参与方会看到 B 和 C 两个签署方，增加理解成本。
        
    *   逻辑冗余：在静默授权下，C 节点的“秒签”显得多余且无法体现“代理”关系。
        

方案 2：作为“授权签署方”加入流程（新增特定角色）

为跨企业授权定制一种特殊的参与方角色。

*   **优点：** 法律语义明确，可以在证据保全报告中清晰界定“代理人（B）”与“被代理主体（C）”。
    
*   **缺点：** 需对现有签署流程引擎进行较大改造，以支持这种“只出场、不操作、自动闭环”的特殊角色。
    

方案 3：作为“影子主体”基于资源关联（推荐方案）

C 不进入签署流程，系统通过合同中引用的“印章 ID”自动建立 C 与合同的关联。

*   **优点：**
    
    *   **流程无侵入：** B 的发起流程保持极简，无需手动添加 C。
        
    *   **权责自洽：** C 的权限基于“印章所有权”自然产生，符合“谁的资源谁能看”的直觉。
        
    *   **多方兼容：** 支持一份合同中包含多个授权方（如 C1、C2）的印章，权限自动隔离。
        
*   **缺点：** 对 PBAC 权限系统的属性穿透能力要求较高，需打通印章资源与合同访问策略。
    

五、 决策推荐方案：影子主体关联模式 (Asset-Based Association)

1. 签署主体认定方案

*   **法律定性：** 认定 **企业 C** 为合同签署主体。
    
*   **实现方式：**
    
    *   底层 CA 证书指纹指向 C。
        
    *   在《证据保全报告》中记录：_“主体 C（证书持有人）之签署行为，由主体 B 基于授权协议（编号：XXX）代为触发执行”_。
        
    *   以此确保签署行为符合《电子签名法》关于“签名人控制”及“代理签署”的合规要求。
        

2. 合同归属与数据策略

*   **存储空间：** 合同物理归属于**发起方 B** 的租户，确保其业务生命周期管理。
    
*   **镜像可见：** C 企业在其租户内的“印章管理-用印记录”中通过关联链接，获得对该合同的 **READ（只读）** 和 **DOWNLOAD（下载）** 权限。
    
*   **管理列表可见：** C企业在其租户内的“企业合同”列表也可以查看并检索与管理此合同。
    
*   **权限保障：** 这种权限是强制性的，不依赖于 B 是否“抄送”，确保 C 的审计权。
    

3. PBAC 权限描述建议

在 PBAC 模型中新增如下策略：

```plaintext
Allow Subject(C_Admin) -> Action (View/Download) -> Resource (Contract_X)
Condition: Contract_X.UsedSeals contains C.Seal_ID AND C.AuthorizationStatus == Active
```

六、 推荐理由

1.  **合规性最优：** 穿透了“视觉印章”与“底层证书”，在静默授权下建立了最强的证据链（授权书 + 资源关联 + 审计日志）。
    
2.  **扩展性最强：** 该模型不仅适用于 FESCO 场景，未来可无缝扩展至集团统筹签署、第三方代办签署等复杂商业场景。
    
3.  **用户感知最小化：** 解决了“谁该进流程、谁不该进流程”的争论，让签署流程回归业务本身，而将管理与审计回归到资源管理侧。
    
    七、 决策结论：
    
    （1）采用方案三的影子授权方案，需要签署支持一个新的旁路租户扩散分支，此分支的特点：
    
    *   与原有由于角色加入导致的租户扩散是平行关系，不是共用。
        
    *   目前仅能通过跨企业印章触发，依据跨企业印章的主体作为旁路扩散的目标
        
    *   被扩散的租户目前从业务上来看只是会读取同步，而不会进行主动的写以及扩散通知
        
    *   跨企业印章扩散的资源没有明确归属与参与方概念，不会受到租户内部成员转交影响
        
    *   跨企业印章资源，可支持查看、下载、备注、关联合同、续签标记等当前租户内的管理能力
