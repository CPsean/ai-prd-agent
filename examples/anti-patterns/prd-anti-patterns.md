## AP-01 模糊的验收标准

### 问题描述
验收标准无法转化为测试用例，QA 不知道怎么判断通过/不通过。

### 反面示例
```markdown
## 验收标准
- [ ] AC-1：页面加载速度快
- [ ] AC-2：用户体验流畅
- [ ] AC-3：数据展示正确
```

### 问题批注
- `页面加载速度快`：没有量化指标，无法测试
- `用户体验流畅`：主观描述，不同人判断标准不同
- `数据展示正确`：范围不明确，"正确"的定义是什么？

### 正确写法
```markdown
## 验收标准
- [ ] AC-1：列表页首屏加载时间 ≤ 2 秒（网络环境：4G，P95）
- [ ] AC-2：列表滚动帧率 ≥ 55fps，无明显卡顿（测试设备：中端 Android）
- [ ] AC-3：列表展示的订单数量与接口返回的 total 字段一致，误差为 0
```

## AP-02 用户故事的 So that 写成了操作步骤

### 问题描述
`So that` 应该描述业务价值或用户目标，而不是操作行为的延续。

### 反面示例
```markdown
As a 用户，
I want to 点击"导出"按钮，
So that 我可以下载文件。
```

### 问题批注
- `So that 我可以下载文件` 是操作结果，不是业务价值
- 读完这句话，不知道用户为什么需要下载文件

### 正确写法
```markdown
As a 财务人员，
I want to 将本月账单导出为 Excel，
So that 我可以在公司财务系统中完成报销归档，无需手动录入数据。
```

## AP-03 边界条件只写正常流程

### 问题描述
PRD 只描述"一切顺利时"的流程，忽略异常、边界、空状态场景，导致开发自行决策，测试发现漏洞。

### 反面示例
```markdown
## 页面 & 交互说明

用户点击"提交"按钮后，系统保存数据并跳转到成功页面。
```

### 问题批注
- 未说明：表单校验失败时如何处理
- 未说明：接口超时或失败时如何处理
- 未说明：重复提交如何处理
- 未说明：数据为空时页面展示什么

### 正确写法
```markdown
## 页面 & 交互说明

**正常流程**：用户点击"提交" → 前端校验通过 → 调用接口 → 成功 → 跳转成功页

**边界条件 & 异常处理**：

| 场景 | 处理方式 |
|------|----------|
| 必填项为空 | 表单内联报错提示，按钮不可点击 |
| 接口返回失败 | Toast 提示错误原因，停留在当前页，按钮恢复可点击 |
| 接口超时（>10s） | Toast 提示"网络超时，请重试"，不跳转 |
| 用户快速重复点击提交 | 第一次点击后按钮置灰，直到接口返回 |
```

## AP-04 字段清单缺少业务规则

### 问题描述
字段清单只列了名称和类型，没有说明取值范围、格式约束、业务含义，开发需要猜测或反复确认。

### 反面示例
```markdown
| 字段名 | 标识符 | 类型 | 必填 |
|--------|--------|------|------|
| 手机号 | phone | string | 是 |
| 金额 | amount | number | 是 |
| 状态 | status | string | 否 |
```

### 问题批注
- `手机号`：格式是什么？支持国际号码吗？
- `金额`：单位是元还是分？精度几位小数？上限是多少？
- `状态`：有哪些枚举值？每个值的含义是什么？

### 正确写法
```markdown
| 字段名 | 标识符 | 类型 | 必填 | 说明 | 业务规则 |
|--------|--------|------|------|------|----------|
| 手机号 | phone | string(20) | 是 | 用户手机号 | 仅支持中国大陆 11 位手机号，格式：1[3-9]XXXXXXXXX |
| 金额 | amount | decimal(10,2) | 是 | 订单金额，单位：元 | 范围 0.01 ~ 99999.99，精确到分 |
| 状态 | status | enum | 否 | 订单状态 | pending=待支付；paid=已支付；cancelled=已取消；默认 pending |
```

## AP-06 AC 混入运营指标

### 问题描述
验收标准（AC）中出现上线后才能观测的运营/业务指标，导致开发/测试无法在交付时判断是否通过，且这些目标与功能本身的完成度无关。

### 反面示例
```markdown
## 验收标准
- [ ] AC-1：账单对账类工单量上线后 1 个月统计下降 ≥ 30%
- [ ] AC-2：业务发起端共 9 个字段空值率 = 0%
- [ ] AC-3：企业用户场景下，发起部门名称空值率 = 0%
```

### 问题批注
- `上线后 1 个月下降 ≥ 30%`：时间窗口型指标，上线当天无法验证，不是 AC
- `空值率 = 0%`：这是数据质量目标，需要用户实际使用后统计，不是功能验收条件
- 将运营指标混入 AC，会让测试无法关闭测试用例，也会让开发质疑自己的交付标准

### 正确写法

将两类内容拆分到各自章节：

```markdown
## 验收标准
- [ ] AC-1：发起工单时，9 个必填字段（含发起部门名称）若为空，前端提交按钮不可点击，并展示内联校验提示
- [ ] AC-2：企业用户身份下，"发起部门名称"字段为必填项，接口层同步校验，空值请求返回 400 并提示具体字段名
- [ ] AC-3：工单提交成功后，系统自动归入对账类工单队列，审核员可在工单列表筛选到该记录

## 成功指标
| 指标 | 目标值 | 观测时间 | 数据来源 |
|------|--------|----------|----------|
| 账单对账类工单量 | 较上线前下降 ≥ 30% | 上线后 1 个月 | 工单系统报表 |
| 发起端 9 个字段空值率 | = 0% | 上线后 2 周稳定期后 | 数据库字段统计 |
```

### 判断口诀
> 上线当天测试能判断通过/不通过 → 放 AC  
> 上线后一段时间观测才知道结果 → 放成功指标

## AP-05 开放问题留而不决

### 问题描述
`approved` 状态的 PRD 中仍有未解决的开放问题，导致开发启动后遭遇需求变更。

### 反面示例
```markdown
## 开放问题

| # | 问题 | 提出方 | 状态 |
|---|------|--------|------|
| 1 | 超时时间定为多少秒？ | 开发 | 待确认 |
| 2 | 这个功能是否需要支持国际化？ | 产品 | 待确认 |
```
（PRD 状态已标注为 `approved`）

### 问题批注
- 超时时间影响接口设计，未确认不能开发
- 国际化影响文案和布局，未确认影响工作量估算

### 正确处理方式
- PRD 状态推进到 `approved` 前，开放问题必须全部关闭或移入正文
- 若确实无法决策，明确标注"决策推迟，当前按 X 方案实施，后续变更走 `/update-prd`"
