# 命令：新建 PRD

根据以下参数新建一个 PRD：

$ARGUMENTS

## 解析参数

从 `$ARGUMENTS` 中识别：
- **类型**：`story-card`、`feature`、`iteration` 或 `epic`（必须，无则询问用户）
- **标题**：中文标题（必须，无则询问用户）

## 执行步骤

**0. 迭代类型确认（仅 feature 类型时执行）**

类型为 `feature` 时，询问用户：

> 这是新功能还是对已有功能的迭代优化？
> - **A. 新功能**：全新能力，使用完整 feature PRD 模板
> - **B. 迭代优化**：对现有功能的调整/扩展/修复，使用精简迭代模板
>
> 请回复 A 或 B。

选 **A**：继续使用 `templates/feature-prd.md`，正常执行后续步骤。

选 **B**：
- 追加询问：「请提供被迭代功能的名称或 PRD ID（如不确定可跳过，后续手动补充）」
- 记录原始 PRD 信息（标题 + ID），后续步骤中用 `templates/iteration-prd.md` 代替 `feature-prd.md`
- 填充 YAML 的 `modifies` 和 `modifies-title` 字段

类型直接为 `iteration` 时，视同选 B，直接询问被迭代的原始功能。
类型为 `story-card` 或 `epic` 时，跳过本步骤。

**0.5 读取作者信息**

检查 `context/workspace-config.md` 是否存在，以及 `author` 字段是否已填写（非占位符 `your-name` 且非空）：

- **已填写**：读取 `author` 字段值，后续步骤中用于填充 YAML frontmatter，静默继续，不打扰用户。
- **未填写或文件不存在**：在执行任何文件操作前，向用户提示：

  > **首次使用提示：** 请告诉我你的姓名或花名，将自动保存到 `context/workspace-config.md`，之后所有 PRD 的 `author` 字段会自动填写，无需每次手动输入。
  >
  > 请输入你的姓名（或回复"跳过"以留空）：

  用户回复后：
  - 若用户提供了名字：创建或更新 `context/workspace-config.md`，将 `author` 字段设为用户输入的值，然后继续。
  - 若用户回复"跳过"：`author` 字段留空，继续执行，不再追问。

**1. 确定 ID**
- 读取 `prds/_registry.md`，找到当前对应类型的最大 ID 编号
- 新 ID = 最大编号 + 1（格式：SC-001 / F-001 / E-001，三位补零）

**2. 冲突检测**

依次检查 `prds/[标题]/` 和 `drafts/[标题]/` 目录是否已存在：
- **任一已存在**：停止执行，告知用户在哪个区域（prds/ 或 drafts/）发现同名 PRD（输出其 ID 和当前状态），并提示：
  - 若要修改内容 → 使用 `/update-prd [标题]`
  - 若要新建同类但不同名的需求 → 请提供不同标题重新运行
  - 不得覆盖已有文件
- **均不存在**：继续执行

**3. 创建文件夹和文件**

创建目录 `drafts/[标题]/`，在其中创建：

a. `prd.md` — 从对应模板复制：
   - story-card → `templates/story-card.md`
   - feature（新功能）→ `templates/feature-prd.md`
   - feature（迭代优化）/ iteration → `templates/iteration-prd.md`
   - epic → `templates/epic-prd.md`

   替换模板占位符：
   - `SC-XXX` / `F-XXX` / `E-XXX` → 新分配的 ID
   - `[标题]` → 用户提供的标题
   - `YYYY-MM-DD` → 今天日期
   - `author:` → 步骤 0.5 读取到的作者名（如已获取）；若用户选择跳过则留空

b. `CHANGELOG.md` — 内容如下：
```markdown
# 变更记录 — [标题]

> AI 读取指南：此文件记录历史变更。查阅需求当前状态请读 `prd.md`，不要读此文件作为主要参考。

---

## v1.0 — [今天日期]

**变更类型**：初始创建

**变更内容**：PRD 初始版本创建。
```

c. `archive/` — 创建空文件夹（放置 `.gitkeep` 占位）

d. 如类型为 `feature` 或 `epic`，判断是否需要创建 `fields.md`：

**需要创建的条件**（满足任意一条）：
- 需求描述中明确提到新增字段、数据存储、数据库表、配置项、API 参数
- 内容填充阶段（步骤 5）生成的 `prd.md` 字段清单章节包含实际字段内容（非空表格）

**不创建的情况**：
- 纯交互 / 体验 / 文案类需求，无新增数据字段
- 字段清单章节为空，即本功能不引入任何新的存储结构

**创建时内容如下：**
```markdown
# 字段清单 — [标题]

> 关联 PRD：[prd.md](prd.md)

## [模块名称]

| 字段名 | 标识符 | 类型 | 长度 | 必填 | 默认值 | 说明 | 业务规则 |
|--------|--------|------|------|------|--------|------|----------|
| | | | | | | | |
```

**4. 暂不更新 `prds/_registry.md`**

PRD 处于草稿阶段，尚未进入正式区，注册表暂不更新。确认移入 `prds/` 后再注册（见步骤 7）。

**5. 内容填充（自动判断）**

完成文件结构创建后，判断是否有足够上下文自动填写 `prd.md` 内容：

**有内容可填的条件**（满足任意一条）：
- `drafts/[标题]/rdd.md` 存在（由 `/requirement-clarifier` 生成）——**优先使用**
- `$ARGUMENTS` 中包含需求描述（不只是类型 + 标题）
- `drafts/` 目录下存在与标题相关的方案设计文档

**有上下文时**：直接基于已有信息填写 `prd.md` 各章节；如判断需要创建 `fields.md`（见步骤 3d），同步填入字段内容。填写完成后在 CHANGELOG.md 中注明"初始版本，由 AI 基于[来源]生成"。

**无上下文时**：保留空模板，告知用户可通过以下方式继续：
- 直接描述需求，让 AI 填写当前 PRD
- 先运行 `/requirement-clarifier` 澄清需求再填写

**5.5 PRD 细节澄清**（仅当草稿内容已填充时执行）

草稿填充完成后，读取 `prd.md` 当前内容，逐维度检查是否存在产品决策尚未明确的点。

**澄清维度**（只问影响方案落笔的关键点，跳过技术实现细节）：
1. **操作入口与反馈**：核心操作从哪里触发？成功/失败后界面如何响应？
2. **角色与数据范围**：不同角色能看什么、能做什么？数据隔离规则？
3. **核心对象状态**：主要业务对象有哪些状态？各状态下允许/禁止哪些操作？
4. **边界与异常场景**：空数据、无权限、操作冲突等情况如何处理？
5. **通知节点**：哪些关键时刻需要通知用户？通知给谁？
6. **批量操作**：是否需要批量处理？部分失败如何响应？

**提问规则**：
- 每轮只问最关键的 ≤3 个问题，用产品语言表达，简洁可读
- 等用户回答后，同步更新 `prd.md` 对应内容，再评估是否还有遗留模糊点
- 持续循环，直到判断所有关键产品决策已明确

**收敛判断**——以下条件均满足时视为收敛：
- 核心操作路径已明确
- 角色与权限边界已明确
- 主要边界场景已有处理策略
- 无明显影响落笔的遗留问题

收敛后输出：
> **方案细节已收敛，PRD 草稿可以进入评审。**

**6. 输出确认**

告知用户：
- 草稿已创建于 `drafts/[标题]/prd.md`，分配 ID：[ID]
- 是否已自动填写内容（或需要手动补充的章节）

**7. 主动询问是否移入正式区**

输出以下询问：

> PRD 草稿已完成。请选择下一步：
> - **A. 留在草稿区**：内容还需继续完善，稍后再确认
> - **B. 移入正式区（prds/）**：内容已完整，可以进入评审流程
>
> 请回复 A 或 B。

根据用户选择：

**选 A**：不做任何修改。告知用户随时可以说「确认 [标题] PRD 移入正式区」触发移入操作。

**选 B**：执行以下操作：
1. 将 `drafts/[标题]/` 整个目录移动到 `prds/[标题]/`
2. 在 `prds/_registry.md` 的"活跃 PRD"表格末尾追加：
   ```
   | [ID] | [标题] | [类型] | draft | prds/[标题]/prd.md | [今天日期] |
   ```
3. 告知用户：PRD 已注册至 `prds/`，状态为 `draft`，可运行 `/prd-summary [标题]` 开始评审准备。

4. **判断是否涉及页面变更**：读取 `prd.md` 内容，判断本需求是否需要生成页面规格和原型。

   **涉及页面变更的判断依据**（满足任意一条）：
   - PRD 描述了新增页面、新增入口、新增弹窗/抽屉
   - PRD 描述了现有页面的布局、组件或操作流程调整
   - PRD 的"页面 & 交互说明"章节有实质内容

   **涉及页面变更时**，输出：
   > **本需求涉及页面变更（[一句话说明：如"新增审批列表页和详情抽屉"]）。**
   > 建议按顺序完成：
   > 1. 运行 `/generate-page-spec [标题]` 生成页面规格卡
   > 2. 规格卡确认后，运行 `/generate-prototype [标题]` 生成可交互原型

   **不涉及页面变更时**，输出：
   > **本需求为纯逻辑/配置类变更，无需生成页面规格和原型，PRD 流程完成。**

   > 如判断有误，可直接告知并手动运行 `/generate-page-spec [标题]`。

> **自然语言触发**：用户在任何时候说「确认 [标题] 移入正式区」或「[标题] PRD 可以了」，均执行上述选 B 的操作。
