# 命令：新建 PRD

根据以下参数新建一个 PRD：

$ARGUMENTS

## 解析参数

从 `$ARGUMENTS` 中识别：
- **类型**：`story-card`、`feature`、`iteration` 或 `epic`（必须，无则询问用户）
- **标题**：中文标题（必须，无则询问用户）

## 执行步骤

**0. 迭代类型确认（仅 feature 类型时执行）**

类型为 `feature` 时，询问用户：

> 这是新功能还是对已有功能的迭代优化？
> - **A. 新功能**：全新能力，使用完整 feature PRD 模板
> - **B. 迭代优化**：对现有功能的调整/扩展/修复，使用精简迭代模板
>
> 请回复 A 或 B。

选 **A**：继续使用 `templates/feature-prd.md`，正常执行后续步骤。

选 **B**：
- 追加询问：「请提供被迭代功能的名称或 PRD ID（如不确定可跳过，后续手动补充）」
- 记录原始 PRD 信息（标题 + ID），后续步骤中用 `templates/iteration-prd.md` 代替 `feature-prd.md`
- 填充 YAML 的 `modifies` 和 `modifies-title` 字段

类型直接为 `iteration` 时，视同选 B，直接询问被迭代的原始功能。
类型为 `story-card` 或 `epic` 时，跳过本步骤。

**0.5 读取作者信息**

检查 `context/workspace-config.md` 是否存在，以及 `author` 字段是否已填写（非占位符 `your-name` 且非空）：

- **已填写**：读取 `author` 字段值，后续步骤中用于填充 YAML frontmatter，静默继续，不打扰用户。
- **未填写或文件不存在**：在执行任何文件操作前，向用户提示：

  > **首次使用提示：** 请告诉我你的姓名或花名，将自动保存到 `context/workspace-config.md`，之后所有 PRD 的 `author` 字段会自动填写，无需每次手动输入。
  >
  > 请输入你的姓名（或回复"跳过"以留空）：

  用户回复后：
  - 若用户提供了名字：创建或更新 `context/workspace-config.md`，将 `author` 字段设为用户输入的值，然后继续。
  - 若用户回复"跳过"：`author` 字段留空，继续执行，不再追问。

**1. 确定 ID**
- 读取 `prds/_registry.md`，找到当前对应类型的最大 ID 编号
- 新 ID = 最大编号 + 1（格式：SC-001 / F-001 / E-001，三位补零）

**2. 冲突检测**

依次检查 `prds/[标题]/prd.md` 和 `drafts/[标题]/prd.md` 文件是否已存在：
- **任一已存在**：停止执行，告知用户在哪个区域（prds/ 或 drafts/）发现同名 PRD（输出其 ID 和当前状态），并提示：
  - 若要修改内容 → 使用 `/update-prd [标题]`
  - 若要新建同类但不同名的需求 → 请提供不同标题重新运行
  - 不得覆盖已有文件
- **均不存在**：继续执行

> **注意**：`drafts/[标题]/` 目录可能已由 `/requirement-clarifier` 创建（含 `rdd.md`），这不构成冲突，继续正常执行即可。

**3. 创建文件夹和文件**

创建目录 `drafts/[标题]/`，在其中创建：

a. `prd.md` — 从对应模板复制：
   - story-card → `templates/story-card.md`
   - feature（新功能）→ `templates/feature-prd.md`
   - feature（迭代优化）/ iteration → `templates/iteration-prd.md`
   - epic → `templates/epic-prd.md`

   替换模板占位符：
   - `SC-XXX` / `F-XXX` / `E-XXX` → 新分配的 ID
   - `[标题]` → 用户提供的标题
   - `YYYY-MM-DD` → 今天日期
   - `author:` → 步骤 0.5 读取到的作者名（如已获取）；若用户选择跳过则留空

b. `CHANGELOG.md` — 内容如下：
```markdown
# 变更记录 — [标题]

> AI 读取指南：此文件记录历史变更。查阅需求当前状态请读 `prd.md`，不要读此文件作为主要参考。

---

## v1.0 — [今天日期]

**变更类型**：初始创建

**变更内容**：PRD 初始版本创建。
```

c. `archive/` — 创建空文件夹（放置 `.gitkeep` 占位）

d. 如类型为 `feature` 或 `epic`，判断是否需要创建 `fields.md`：

**需要创建的条件**（满足任意一条）：
- 需求描述中明确提到新增字段、数据存储、数据库表、配置项、API 参数
- 内容填充阶段（步骤 5）生成的 `prd.md` 字段清单章节包含实际字段内容（非空表格）

**不创建的情况**：
- 纯交互 / 体验 / 文案类需求，无新增数据字段
- 字段清单章节为空，即本功能不引入任何新的存储结构

**创建时内容如下：**
```markdown
# 字段清单 — [标题]

> 关联 PRD：[prd.md](prd.md)

## [模块名称]

| 字段名 | 标识符 | 类型 | 长度 | 必填 | 默认值 | 说明 | 业务规则 |
|--------|--------|------|------|------|--------|------|----------|
| | | | | | | | |
```

**4. 暂不更新 `prds/_registry.md`**

PRD 处于草稿阶段，尚未进入正式区，注册表暂不更新。确认移入 `prds/` 后再注册（见步骤 7）。

**5. 内容填充（自动判断）**

完成文件结构创建后，首先检查 rdd.md 状态，再判断如何填充内容。

**5-0. rdd.md 状态检查**

若 `drafts/[标题]/rdd.md` 存在，读取其 `status` 字段：
- `status: rdd-complete` → 正常读取，优先用于填充 §3-§7，继续执行
- `status: rdd-in-progress` 或 `story-confirmed` → 输出提示后询问用户：

  > **检测到「[标题]」的需求分析尚未完成（当前状态：[status]）。**
  > 建议先运行 `/requirement-clarifier [标题]` 补全 RDD，PRD 内容将更完整。
  >
  > 或者：
  > - 回复 **A** — 继续基于现有内容起草 PRD（部分章节可能不完整）
  > - 回复 **B** — 先去完成需求分析，稍后再建 PRD
  
  用户选 A 则继续，选 B 则停止执行。

**5-1. 领域检查清单扫描（条件触发）**

检查 `context/domain-checklists.md` 是否存在：
- **不存在**：输出提示「未检测到领域检查清单配置，如需启用可运行 /init 或手动创建 context/domain-checklists.md」，静默跳过
- **存在**：读取注册表（最多 10 行，超出警告并跳过），筛选 `扫描时机` 含 `new-prd` 的行，将需求描述（`$ARGUMENTS` + `rdd.md` 内容）与各行触发关键词逐一匹配：
  - **命中**（按需加载，每次执行最多触发 3 个清单）：读取对应清单文件（建议 ≥ 3 条检查项、≤ 150 行），执行扫描，在开始填充 `prd.md` 内容前输出领域影响结论块（格式：有影响 / 无影响 / 待确认）
  - **未命中**：静默跳过

**有内容可填的条件**（满足任意一条）：
- `drafts/[标题]/rdd.md` 存在且 `status: rdd-complete`——**优先使用**
- `drafts/[标题]/rdd.md` 存在但状态未完成，且用户选择继续（选 A）
- `$ARGUMENTS` 中包含需求描述（不只是类型 + 标题）
- `drafts/` 目录下存在与标题相关的方案设计文档

**有上下文时**：直接基于已有信息填写 `prd.md` 各章节；如判断需要创建 `fields.md`（见步骤 3d），同步填入字段内容。填写完成后在 CHANGELOG.md 中注明"初始版本，由 AI 基于[来源]生成"。

填写各章节时按需读取对应 context 文件（文件不存在时静默跳过）：

| 写到这个章节时 | 按需读取 |
|--------------|---------|
| §4 需求对象与概念模型 | `context/business-glossary.md` — 识别已有术语，避免重复定义 |
| §5 功能结构 / §7 功能清单 | `context/product-feature-map.md` — 获取已有模块前缀，判断新增/调整/复用 |
| §8 权限矩阵（§8.x.7）| `context/permission-model.md` — 填充角色权限矩阵 |
| §9 非功能性需求（§9.4 兼容性）| `context/platform-support.md` — 读取已支持端，生成对应兼容性说明 |

填写 `feature-prd.md` 类型文档时，额外执行以下两项：

**5a. 功能编号生成**（填写 §7 功能清单时）

1. 读取 `drafts/[标题]/rdd.md`（若存在）中的功能模块列表
2. 此步骤触发读取 `context/product-feature-map.md`（若存在）中的功能编号前缀映射表
3. 对每个模块：已有前缀 → 复用；新模块 → 推导英文缩写（2-4 字母全大写），记录待追加到映射表
4. 按模块分配 `[AREA]-[CATEGORY]-001/002/003` 格式编号，填入 §7 功能清单

**5b. §8 逐功能展开**（填写 §8 功能需求说明书时）

对 §7 中每条功能自动生成对应的 §8.x 子节：
- **§8.x.1 任务故事**：始终填写
- **§8.x.2 逻辑实现规范（Context/Action/Outcome）**：始终填写
- **§8.x.3 异常处理要求**：始终填写
- **§8.x.4 业务流转图**：有分支判断时填写 Mermaid，否则填「不涉及」
- **§8.x.5 数据字典**：有新增/修改字段时填写，否则填「不涉及」
- **§8.x.6 状态流转表**：业务对象有生命周期状态时填写，否则填「不涉及」
- **§8.x.7 权限矩阵**：涉及多角色操作时填写（此时触发读取 `context/permission-model.md`），否则填「不涉及」
- **§8.x.8 边界条件与并发规则**：共享资源/并发操作时填写，否则填「不涉及」
- **§8.x.9 PRD-页面规格卡映射**：涉及页面变更时填写，否则填「不涉及」

填写**验收标准（AC）章节**时，严格遵守以下边界：
- **AC 只写**：上线当天测试人员即可判断通过/不通过的功能行为条件（如：字段校验规则、操作后的界面反馈、接口行为、权限控制）
- **AC 不写**：需要上线后一段时间才能观测的运营/业务指标（如："上线 N 天后指标下降 X%"、"空值率 = 0%"、"用户转化率提升"等）
- 将上述运营指标填入模板中的**"成功指标"章节**（含目标值、观测时间、数据来源）

**无上下文时**：保留空模板，告知用户可通过以下方式继续：
- 直接描述需求，让 AI 填写当前 PRD
- 先运行 `/requirement-clarifier` 澄清需求再填写

**5.5 PRD 细节澄清**（仅当草稿内容已填充时执行）

草稿填充完成后，读取 `prd.md` 当前内容，逐维度检查是否存在产品决策尚未明确的点。

**澄清维度**（只问影响方案落笔的关键点，跳过技术实现细节）：
1. **操作入口与反馈**：核心操作从哪里触发？成功/失败后界面如何响应？
2. **角色与数据范围**：不同角色能看什么、能做什么？数据隔离规则？
3. **核心对象状态**：主要业务对象有哪些状态？各状态下允许/禁止哪些操作？
4. **边界与异常场景**：空数据、无权限、操作冲突等情况如何处理？
5. **通知节点**：哪些关键时刻需要通知用户？通知给谁？
6. **批量操作**：是否需要批量处理？部分失败如何响应？

**提问规则**：
- 每轮只问最关键的 ≤3 个问题，用产品语言表达，简洁可读
- 等用户回答后，同步更新 `prd.md` 对应内容，再评估是否还有遗留模糊点
- 持续循环，直到判断所有关键产品决策已明确

**收敛判断**——以下条件均满足时视为收敛：
- 核心操作路径已明确
- 角色与权限边界已明确
- 主要边界场景已有处理策略
- 无明显影响落笔的遗留问题

收敛后输出：
> **方案细节已收敛，PRD 草稿可以进入评审。**

**6. 输出确认**

告知用户：
- 草稿已创建于 `drafts/[标题]/prd.md`，分配 ID：[ID]
- 是否已自动填写内容（或需要手动补充的章节）

**6.1 上下文缺口提醒（CTX-RMD-001）**

检查本次执行过程中是否发现关键 context 文件缺失或为空（如 product-background.md、user-persona.md、business-glossary.md 等），或对话中出现未在 business-glossary.md 定义的业务名词。

有缺口时，在输出末尾追加提醒（不中断主流程）：

> 💡 **上下文补充建议**：发现 [文件名] 尚未初始化 / [术语] 未在术语表中定义。
> 建议运行 `/import-context` 补充，或直接告诉我相关信息我来帮你写入。

同一对话中同一缺口只提醒一次。

**7. 主动询问是否移入正式区**

输出以下询问：

> PRD 草稿已完成。请选择下一步：
> - **A. 留在草稿区**：内容还需继续完善，稍后再确认
> - **B. 移入正式区（prds/）**：内容已完整，可以进入评审流程
>
> 请回复 A 或 B。

根据用户选择：

**选 A**：不做任何修改。告知用户随时可以说「确认 [标题] PRD 移入正式区」触发移入操作。

**选 B**：执行以下操作：
1. 将 `drafts/[标题]/` 整个目录移动到 `prds/[标题]/`
2. 在 `prds/_registry.md` 的"活跃 PRD"表格末尾追加：
   ```
   | [ID] | [标题] | [类型] | draft | prds/[标题]/prd.md | [今天日期] |
   ```
3. 告知用户：PRD 已注册至 `prds/`，状态为 `draft`，可运行 `/prd-summary [标题]` 开始评审准备。

4. **判断是否涉及页面变更**：读取 `prd.md` 内容，判断本需求是否需要生成页面规格和原型。

   **涉及页面变更的判断依据**（满足任意一条）：
   - PRD 描述了新增页面、新增入口、新增弹窗/抽屉
   - PRD 描述了现有页面的布局、组件或操作流程调整
   - PRD 的"页面 & 交互说明"章节有实质内容

   **涉及页面变更时**，输出：
   > **本需求涉及页面变更（[一句话说明：如"新增审批列表页和详情抽屉"]）。**
   > 建议按顺序完成：
   > 1. 运行 `/generate-page-spec [标题]` 生成页面规格卡
   > 2. 规格卡确认后，运行 `/generate-prototype [标题]` 生成可交互原型

   **不涉及页面变更时**，输出：
   > **本需求为纯逻辑/配置类变更，无需生成页面规格和原型，PRD 流程完成。**

   > 如判断有误，可直接告知并手动运行 `/generate-page-spec [标题]`。

5. **context 文件同步提示**（仅 feature 或 epic 类型时执行）：

   a. 读取已移入正式区的 `prd.md` 中 §5 功能结构，提取本 PRD 新增/调整的功能节点和模块英文前缀。
   
   b. **产品功能结构更新**：若存在新增节点或新模块前缀，输出：
   
   > **建议将以下内容追加到 `context/product-feature-map.md`：**
   > - 前缀映射表追加：[模块名] → [英文前缀]（若有新模块）
   > - 功能节点追加：[新节点列表]
   >
   > 回复「确认」写入，或告知需要修改。
   
   用户确认后：更新 `context/product-feature-map.md` 的前缀映射表和 Mermaid 功能树。若文件不存在，提议以本 PRD 内容为基础创建初始文件。
   
   c. **业务术语同步**：若 `/requirement-clarifier` 阶段 Step 6.5 尚未完成术语追加（例如用户跳过了术语确认步骤），此时一并提示，询问是否补充追加到 `context/business-glossary.md`。

> **自然语言触发**：用户在任何时候说「确认 [标题] 移入正式区」或「[标题] PRD 可以了」，均执行上述选 B 的操作。
