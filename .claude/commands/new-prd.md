# 命令：新建 PRD

根据以下参数新建一个 PRD：

$ARGUMENTS

## 解析参数

从 `$ARGUMENTS` 中识别：
- **类型**：`story-card`、`feature` 或 `epic`（必须，无则询问用户）
- **标题**：中文标题（必须，无则询问用户）

## 执行步骤

**1. 确定 ID**
- 读取 `prds/_registry.md`，找到当前对应类型的最大 ID 编号
- 新 ID = 最大编号 + 1（格式：SC-001 / F-001 / E-001，三位补零）

**2. 冲突检测**

检查 `prds/[标题]/` 目录是否已存在：
- **已存在**：停止执行，告知用户该路径已有 PRD（输出其 ID 和当前状态），并提示：
  - 若要修改内容 → 使用 `/update-prd [标题]`
  - 若要新建同类但不同名的需求 → 请提供不同标题重新运行
  - 不得覆盖已有文件
- **不存在**：继续执行

**3. 创建文件夹和文件**

创建目录 `drafts/[标题]/`，在其中创建：

a. `prd.md` — 从对应模板复制：
   - story-card → `templates/story-card.md`
   - feature → `templates/feature-prd.md`
   - epic → `templates/epic-prd.md`

   替换模板占位符：
   - `SC-XXX` / `F-XXX` / `E-XXX` → 新分配的 ID
   - `[标题]` → 用户提供的标题
   - `YYYY-MM-DD` → 今天日期

b. `CHANGELOG.md` — 内容如下：
```markdown
# 变更记录 — [标题]

> AI 读取指南：此文件记录历史变更。查阅需求当前状态请读 `prd.md`，不要读此文件作为主要参考。

---

## v1.0 — [今天日期]

**变更类型**：初始创建

**变更内容**：PRD 初始版本创建。
```

c. `archive/` — 创建空文件夹（放置 `.gitkeep` 占位）

d. 如类型为 `feature` 或 `epic`，判断是否需要创建 `fields.md`：

**需要创建的条件**（满足任意一条）：
- 需求描述中明确提到新增字段、数据存储、数据库表、配置项、API 参数
- 内容填充阶段（步骤 5）生成的 `prd.md` 字段清单章节包含实际字段内容（非空表格）

**不创建的情况**：
- 纯交互 / 体验 / 文案类需求，无新增数据字段
- 字段清单章节为空，即本功能不引入任何新的存储结构

**创建时内容如下：**
```markdown
# 字段清单 — [标题]

> 关联 PRD：[prd.md](prd.md)

## [模块名称]

| 字段名 | 标识符 | 类型 | 长度 | 必填 | 默认值 | 说明 | 业务规则 |
|--------|--------|------|------|------|--------|------|----------|
| | | | | | | | |
```

**4. 暂不更新 `prds/_registry.md`**

PRD 处于草稿阶段，尚未进入正式区，注册表暂不更新。确认移入 `prds/` 后再注册（见步骤 7）。

**5. 内容填充（自动判断）**

完成文件结构创建后，判断是否有足够上下文自动填写 `prd.md` 内容：

**有内容可填的条件**（满足任意一条）：
- `$ARGUMENTS` 中包含需求描述（不只是类型 + 标题）
- `drafts/` 目录下存在与标题相关的方案设计文档

**有上下文时**：直接基于已有信息填写 `prd.md` 各章节；如判断需要创建 `fields.md`（见步骤 3d），同步填入字段内容。填写完成后在 CHANGELOG.md 中注明"初始版本，由 AI 基于[来源]生成"。

**无上下文时**：保留空模板，告知用户可通过以下方式继续：
- 直接描述需求，让 AI 填写当前 PRD
- 先运行 `/requirement-clarifier` 澄清需求再填写

**6. 输出确认**

告知用户：
- 草稿已创建于 `drafts/[标题]/prd.md`，分配 ID：[ID]
- 是否已自动填写内容（或需要手动补充的章节）

**7. 主动询问是否移入正式区**

输出以下询问：

> PRD 草稿已完成。请选择下一步：
> - **A. 留在草稿区**：内容还需继续完善，稍后再确认
> - **B. 移入正式区（prds/）**：内容已完整，可以进入评审流程
>
> 请回复 A 或 B。

根据用户选择：

**选 A**：不做任何修改。告知用户随时可以说「确认 [标题] PRD 移入正式区」触发移入操作。

**选 B**：执行以下操作：
1. 将 `drafts/[标题]/` 整个目录移动到 `prds/[标题]/`
2. 在 `prds/_registry.md` 的"活跃 PRD"表格末尾追加：
   ```
   | [ID] | [标题] | [类型] | draft | prds/[标题]/prd.md | [今天日期] |
   ```
3. 告知用户：PRD 已注册至 `prds/`，状态为 `draft`，可运行 `/prd-summary [标题]` 开始评审准备。

> **自然语言触发**：用户在任何时候说「确认 [标题] 移入正式区」或「[标题] PRD 可以了」，均执行上述选 B 的操作。
