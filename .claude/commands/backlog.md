# 命令：需求池管理

$ARGUMENTS

## 步骤 0：初始化检查

检查 `backlog/requirement-pool.md` 是否存在：

- **存在**：静默继续，进入步骤 1
- **不存在**：自动创建文件，含五个章节表头（新需求/行动项/开放问题/技术债/归档），内容如下：

```markdown
# 需求池

> 需求池是 RDD 前置阶段的需求统一收口区，管理新需求、行动项、开放问题、技术债四种类型条目。
> 行序即为精确优先级（P0-P3 为粗分标签），修改行序即修改优先级。

---

## 新需求

| 序号 | ID | 标题 | 优先级 | 状态 | 来源 | 关联PRD | 创建日期 |
|------|-----|------|--------|------|------|---------|----------|

---

## 行动项

| 序号 | ID | 内容 | 关联PRD | 状态 | 创建日期 |
|------|-----|------|---------|------|----------|

---

## 开放问题

| ID | 问题 | 关联PRD | 状态 | 结论 | 创建日期 |
|-----|------|---------|------|------|----------|

---

## 技术债

| 序号 | ID | 内容 | 优先级 | 状态 | 来源 | 创建日期 |
|------|-----|------|--------|------|------|----------|

## 归档

| ID | 类型 | 标题/内容 | 最终状态 | 归档日期 | 归档原因 |
```

创建后输出：

> **需求池已创建。** 是否扫描已有 PRD 导入待办和开放问题？
> - 回复 **是** → 执行全量扫描（步骤 6-2）
> - 回复 **跳过** → 继续执行你的原始请求

PM 选择扫描 → 执行步骤 6-2 后回到步骤 1 处理原始意图。
PM 选择跳过 → 直接进入步骤 1。

## 步骤 1：意图路由（BKL-RTG-001）

解析 `$ARGUMENTS` 中的自然语言，按以下优先级匹配意图：

| 意图 | 触发词/模式 | 分发步骤 |
|------|------------|---------|
| 录入 | "有个需求"/"记一下"/"加一条"/"有个想法"/"记录" + 需求描述 | → 步骤 2 |
| 查看 | "看一下"/"目前有哪些"/"列出"/"需求池"/"待办列表" | → 步骤 3 |
| 排序 | "排到…前面"/"移到第…位"/"调整顺序"/"优先级调整"/"置顶"/"置底" | → 步骤 4 |
| 状态更新 | "把…改为…"/"…已完成"/"关闭…"/"归档…" | → 步骤 5 |
| 扫描（指定 PRD） | "扫一下"/"扫描"/"提取待办"/"检查…的TODO" + PRD ID/标题 | → 步骤 6-1 |
| 排期 | "排个期"/"排期"/"优先级排期"/"去排期" | → 步骤 7 |
| 初始化 | "初始化需求池"/"全量扫描"/"导入所有待办" | → 步骤 6-2 |
| 澄清 | "[条目ID] 可以开始澄清了"/"直接澄清"/"马上分析"/"现在就搞清楚" | → 步骤 8 |

**匹配规则**：
- 匹配成功 → 执行对应步骤
- 同时匹配多个意图 → 展示候选意图（≤3个），请 PM 选择
- 匹配失败 → 展示可用操作列表：

> `/backlog` 支持以下操作：
> - **录入**：记录新需求/行动项/开放问题/技术债（如 `/backlog 有个需求：批量导出`）
> - **查看**：查看需求池内容（如 `/backlog 看一下需求池`）
> - **排序**：调整条目优先级（如 `/backlog 把 REQ-002 排到 REQ-001 前面`）
> - **扫描**：从 PRD 中提取待办（如 `/backlog 扫一下 F-012 的待办`）
> - **排期**：将需求送入排期流程（如 `/backlog 拿新需求去排个期`）
> - **初始化**：全量扫描已有 PRD 导入待办
>
> 请描述你要做的操作。

## 步骤 2：录入流程（BKL-ITEM-001）

### 2-1. 类型判断

基于 PM 输入的自然语言判断条目类型：

| 判断信号 | 归类为 |
|---------|-------|
| 描述新功能/新能力/用户反馈/改进建议 | 新需求 |
| 提及具体 PRD 编号 + 待完成事项（"还没补"/"待补充"/"TODO"） | 行动项 |
| 以疑问形式描述（"是否需要"/"要不要"/"怎么处理"）+ 关联 PRD | 开放问题 |
| 提及性能/架构/代码质量/技术优化 | 技术债 |
| 无法判断 | 展示四种类型说明，请 PM 选择 |

无法判断时展示：

> 请选择条目类型：
> - **新需求**：新功能/新能力/用户反馈/改进建议（所有字段均可选）
> - **行动项**：某个 PRD 的待完成事项（需关联 PRD）
> - **开放问题**：待讨论决策的问题（需关联 PRD）
> - **技术债**：性能/架构/代码质量等技术优化项

### 2-2. 产品线检查（BR-05）

读取 `context/product-background.md`（不存在时跳过此检查），对比产品定位。若输入内容明显不属于当前产品线（无法与产品核心能力或用户场景关联）：

> 该需求似乎不属于 [产品名] 产品线，请确认是否录入。

PM 坚持 → 继续录入。PM 取消 → 终止。

### 2-3. 补充字段

按类型引导补充（非强制，PM 可直接确认跳过）：

- **新需求**：来源（选填）、优先级 P0-P3（默认 P2）、关联PRD（选填）
- **行动项**：关联PRD（必填，PM 未提供时提示"行动项需关联 PRD，请提供 PRD ID 或改为录入新需求"）
- **开放问题**：关联PRD（必填，规则同行动项）
- **技术债**：来源（选填）、优先级 P0-P3（默认 P2）

**最小输入模式**：PM 只提供标题/描述即可提交，AI 按以下兜底：
- 类型无法判断 → 默认「新需求」
- 优先级未指定 → 默认 P2
- 来源未指定 → 留空

### 2-4. 重复检测

写入前检查已有条目，若发现标题/内容高度相似（关键词重叠 ≥ 80%）：

> 发现相似条目 [ID]：[标题]，是否仍要新建？

PM 确认 → 写入。PM 拒绝 → 取消。

### 2-5. 写入

PM 确认后：
1. 分配 ID：读取对应章节当前最大编号 + 1（REQ-001 / ACT-001 / OQ-001 / TD-001，三位补零）
2. 设定默认状态：新需求/技术债 →「收集」，行动项 →「待处理」，开放问题 →「待决策」
3. 创建日期 = 今天
4. 追加到对应章节末尾（新需求/行动项/技术债的序号列自动生成）
5. 输出确认："已录入 [ID]：[标题]，类型：[类型]，状态：[状态]"

## 步骤 3：查看流程（BKL-ITEM-002）

读取 `backlog/requirement-pool.md`，解析 PM 的查看意图：

- **无筛选条件**（"看一下需求池"）：输出全部四个章节的条目概览，**不含归档区**。标注各章节条目数量。
- **按类型**（"看新需求"/"有哪些技术债"）：仅输出对应章节
- **按状态**（"有哪些待处理的"）：跨章节筛选指定状态的条目
- **按关联PRD**（"F-012 还有哪些待办"）：跨章节筛选关联指定 PRD 的条目

**输出格式**：
- 表格头部标注筛选条件和命中条数
- 归档区默认不展示，PM 明确要求（"看归档"/"看已关闭的"）才展示

**异常处理**：
- 需求池为空 → 提示"需求池为空，运行 `/backlog` 并输入需求描述开始录入"
- 筛选无结果 → 提示"未找到匹配条目"，展示各类型条目数量

## 步骤 4：排序流程（BKL-ITEM-003）

### 4-1. 排序前置检查

解析 PM 指令，识别条目 ID 和目标位置：
- "排到 XX 前面" / "移到第 N 位" / "置顶" / "置底"

根据条目 ID 前缀判断所属章节：
- REQ- → 新需求 ✅
- ACT- → 行动项 ✅
- TD- → 技术债 ✅
- OQ- → 开放问题 ❌ → 提示"开放问题章节不支持排序"，终止

### 4-2. 执行排序

1. 在对应章节中调整行序
2. 重排序号列（从 1 开始连续递增）
3. 写入文件
4. 输出调整后的章节表格
5. 提示"已将 [ID] 移至第 [N] 位"

**异常处理**：
- 条目 ID 不存在 → 提示"未找到 [ID]，请确认编号"
- 目标位置超出范围 → 自动置底，提示"已移至末尾（第 N 位）"

## 步骤 5：状态更新与归档流程（BKL-ITEM-004）

### 5-1. 状态更新

解析 PM 指令，识别条目 ID 和目标状态。

**各类型允许的状态流转**（BR-04）：

| 类型 | 状态链 |
|------|-------|
| 新需求/技术债 | 收集 → 待澄清 → RDD中 → PRD草稿 → PRD已完成 → 已关闭 |
| 行动项 | 待处理 → 进行中 → 已完成 |
| 开放问题 | 待决策 → 已决策（需提供结论） |

更新条目状态字段，写入文件。

### 5-2. 终态归档提议

若新状态为终态（已完成/已关闭/已决策），提议归档：

> 条目 [ID] 已达终态（[状态]），建议移入归档区。确认归档？

PM 确认后：
1. 从原章节删除该行
2. 重排原章节序号
3. 在「归档」章节追加一行：原始 ID / 类型 / 标题或内容 / 最终状态 / 归档日期（今天）/ 归档原因
4. 输出"[ID] 已归档，原因：[原因]"

PM 拒绝 → 仅更新状态，不归档。

## 步骤 6-1：指定 PRD 扫描（BKL-SCAN-001）

### 6-1-1. 定位 PRD

从 PM 输入中识别 PRD ID 或标题。读取 `prds/_registry.md` 定位 PRD 路径。

- PRD 不存在 → 提示"未找到 [ID] 对应的 PRD"，终止

### 6-1-2. 扫描提取

读取目标 PRD 的 `prd.md`，提取：
- §12 开放问题表格中状态为「待确认」的条目 → 归类为「开放问题」
- 正文中的 TODO / 待补充 / 待确认标记 → 归类为「行动项」

### 6-1-3. 去重比对

与 `backlog/requirement-pool.md` 现有条目比对（按内容相似度），标注"新发现"/"已存在"。

### 6-1-4. 提议入池

逐条列出新发现的条目，每条标注建议类型和关联 PRD：

> **扫描 [PRD ID] 结果**：发现 N 条新条目
>
> 1. [类型] [内容摘要] — 新发现
> 2. [类型] [内容摘要] — 已存在（[ID]）
>
> 确认录入新发现的条目？（逐条确认或回复"全部录入"）

PM 确认的条目：分配 ID，关联PRD 字段自动填入扫描来源 PRD ID，写入对应章节。

输出摘要："扫描 [PRD ID] 完成，发现 N 条，入池 M 条"

**异常处理**：
- 无新发现 → 提示"未发现新的行动项或开放问题"

## 步骤 6-2：全量初始化扫描（BKL-SCAN-002）

1. 读取 `prds/_registry.md`，获取所有活跃 PRD 列表
   - 无已注册 PRD → 提示"暂无已注册的 PRD，无法执行初始化扫描"，终止
2. 逐个执行步骤 6-1 的扫描逻辑
3. 汇总所有发现的条目，按 PRD 和类型分组展示
4. PM 可选择"全部录入"或逐条确认
5. 输出摘要："全量扫描完成，共扫描 N 个 PRD，发现 M 条新条目，已入池 K 条"

## 步骤 7：排期衔接（BKL-PLAN-001）

1. AI 识别 PM 要排期的范围（"拿新需求去排个期"/"技术债也排一下"/"都排"）
2. 提取对应章节的**活跃条目**（排除已关闭/归档）
3. 展示即将送入排期的条目列表：

> **即将送入排期的条目**（共 N 条）：
>
> | ID | 标题/内容 | 优先级 | 状态 |
> |-----|----------|--------|------|
> | REQ-001 | ... | P1 | 收集 |
>
> 确认后进入排期流程。

4. PM 确认后衔接 roadmap-planning 技能

**异常处理**：
- 无活跃条目 → 提示"当前无活跃的[新需求/技术债]，无法排期"

---

## 步骤 8：澄清衔接

当 PM 明确表达澄清意图（"REQ-001 可以开始澄清了"/"直接澄清"）时：

1. 识别目标条目 ID（如有）
2. 提议更新该条目状态为「待澄清」（PM 确认后写入）
3. 提取条目标题/内容
4. 衔接 `/requirement-clarifier [条目标题/内容]`

> **重要规则（BR-01）**：`/backlog` 默认将输入录入需求池，**不自动触发** `/requirement-clarifier`。仅当 PM 明确表达澄清意图时才衔接。判断标准：PM 输入中必须包含"澄清"/"分析"/"搞清楚"/"开始 clarify"等明确的澄清动作词，而非仅仅描述需求内容。

## 飞轮触发规则

以下飞轮动作不在 `/backlog` 命令中执行，而是在对应命令的执行流程中触发。此处列出判断逻辑供参考：

### BKL-FLY-001：PRD 移入正式区后扫描

触发点：`/new-prd` 步骤 7 选 B 执行完成后（context 同步提议之后）

1. 检查 `backlog/requirement-pool.md` 是否存在（不存在则跳过，不自动创建）
2. 读取刚移入正式区的 PRD 的 §12 和正文 TODO
3. 有新条目 → 输出需求池同步提议块，逐条列出，等待 PM 确认
4. 无 TODO/OQ → 静默跳过

### BKL-FLY-002：PRD 更新后扫描增量

触发点：`/update-prd` 完成后

1. 检查 `backlog/requirement-pool.md` 是否存在（不存在则跳过）
2. 对比更新前后内容（利用 archive 中旧版本），识别新增 TODO/OQ
3. 与需求池比对去重
4. 有新增 → 输出增量入池提议
5. 无新增 → 静默跳过

### BKL-FLY-003：关联状态自动提议

触发点：`/requirement-clarifier` RDD 完成、`/new-prd` 创建草稿、`/new-prd` 移入正式区

1. 检查 `backlog/requirement-pool.md` 是否存在（不存在则跳过）
2. 读取需求池，查找标题与当前 RDD/PRD 标题匹配的条目
3. 有匹配 → 提议更新状态（如 收集→待澄清、待澄清→RDD中、RDD中→PRD草稿、PRD草稿→PRD已完成）
4. 无匹配 → 静默跳过

**强制规则（BR-02）**：以上所有飞轮动作，AI 必须先输出提议，PM 确认后才执行写入，不可静默写入。
