# 命令：PRD 问答

基于 PRD 及 context 知识库，回答关于需求的具体问题：

$ARGUMENTS

## 使用说明

任何人（产品、研发、测试、设计）都可以用此命令提问，例如：
- `/prd-qa 用户登录模块的 Session 超时时间是多少？`
- `/prd-qa F-001 的验收标准第二条具体是什么？`
- `/prd-qa 这个功能的字段清单在哪里？`
- `/prd-qa 这个产品的定位是什么？`

## 执行步骤

**1. 意图识别（KQA-QRY-001）**

分析 `$ARGUMENTS` 中的关键词，识别问题领域：

| 领域 | 信号关键词 | 优先检索层 |
|------|-----------|-----------|
| 产品背景类 | "是什么"/"定位"/"背景"/"有哪些功能" | context/ 优先 |
| 功能规则类 | "规则"/"逻辑"/"流程"/"怎么处理" | context/ 优先，未命中进入 PRD 层 |
| 字段定义类 | "字段"/"参数"/"类型"/"长度"/"格式" | context/ 优先，未命中进入 PRD 层 |
| 边界条件类 | "边界"/"异常"/"如果…怎么"/"失败时" | context/ 优先，未命中进入 PRD 层 |
| 权限规则类 | "权限"/"角色"/"谁能"/"谁看" | context/ 优先，未命中进入 PRD 层 |

**意图处理规则**：
- **意图明确**：静默确认领域，直接进入分层检索（不向用户展示中间步骤）
- **意图模糊**（无法判断具体功能或领域）：输出一句话确认——"您是在问[方向A]还是[方向B]？"，等待用户回复后再检索。**最多追问一次**，不循环追问
- **纯闲聊**（不含任何产品相关关键词）：提示"这个问题超出了 PRD 知识库的范围，请提问与产品需求相关的问题"，不进入检索
- **未指定目标 PRD**：询问"请问是关于哪个功能的问题？"，不随意选择一个 PRD 回答
- **多领域问题**（如"[字段]的权限规则是什么"）：按优先级先检索主领域，未命中再检索次领域

**2. 分层定向检索（KQA-QRY-002）**

按以下四层优先级顺序逐层检索，**命中即停，不继续向下**：

**第一层：context/**

根据问题领域读取对应 context 文件：
- 产品背景 → `context/product-background.md`
- 权限规则 → `context/permission-model.md`
- 业务术语 → `context/business-glossary.md`
- 用户角色 → `context/user-persona.md`
- 产品策略 → `context/product-strategy.md`
- 功能结构 → `context/product-feature-map.md`

命中 → 输出答案 + 来源标注（见步骤 3），停止检索。

**第二层：正式区 PRD 索引**

读取 `prds/_registry.md`，根据问题中的 PRD ID 或功能名称关键词定位相关 PRD：
- 包含 PRD ID（如 F-001）→ 按 ID 匹配
- 包含功能名称关键词 → 按标题匹配
- 未找到相关 PRD → 跳过第三层，进入第四层

**第三层：具体 prd.md**

读取第二层定位的 PRD（`prds/[标题]/prd.md`），如有 `fields.md` 一并读取。精确查找问题相关章节。

- 只读 `prd.md` 当前版本，不读 `archive/` 下的历史文件
- 每次检索最多读取 3 份 PRD；超出时提示"相关 PRD 较多，已检索前 3 份，如未命中请指定具体 PRD 标题重新提问"

命中 → 输出答案 + 来源标注，停止检索。

**第四层：草稿区 drafts/**

检索 `drafts/` 目录下相关标题的 `prd.md` 或 `rdd.md`。

- 命中 → 输出答案 + **⚠️ 草稿警告标注**（见步骤 3）
- 未命中 → 触发兜底回复（见步骤 4）

**3. 答案生成与来源标注（KQA-ANS-001）**

**回答规则**：
- 基于原文生成自然语言回答，**不添加推断内容**
- 直接给出答案，不大段复述 PRD 原文
- 引用原文不超过 300 字，超出时摘取最相关段落，标注"以下为节选"
- 引用原文时使用引号

**来源标注格式**（回答末尾追加）：

正式区 PRD 来源：
```
> 来源：prds/[标题]/prd.md §[章节编号] [章节名]
```

context 来源：
```
> 来源：context/[文件名].md
```

草稿区来源：
```
> ⚠️ 来源：drafts/[标题]/prd.md（草稿文档，未正式发布）
```

**特殊场景**：
- **命中章节内容为空/占位符**：回复"该章节在 PRD 中存在但内容未填写，建议通过 `/update-prd` 补充"
- **多个 PRD 对同一概念有不同定义**：提示"发现多处相关定义，请确认以哪个 PRD 为准"，列出冲突的两处原文及各自来源
- **PRD 内容存在歧义**：输出原文，标注"此处表达可能存在歧义，建议 PM 通过 `/update-prd` 补充澄清"

**4. 兜底回复与补充引导（KQA-QRY-003）**

四层检索均未命中时，按以下格式输出：

**标准兜底声明**（不做任何推断）：
```
当前文档中暂无关于「[问题关键词]」的记录。
```

**补充建议**（根据问题类型选择最相关的一条）：
- 有相关 PRD 但章节缺失 → "建议通过 `/update-prd [标题]` 补充相关内容"
- 连相关 PRD 都不存在 → "建议先通过 `/requirement-clarifier` 澄清需求，再建立 PRD"
- 问题属于产品背景/策略层面 → "建议在 `context/product-background.md` 或 `context/product-strategy.md` 中补充"
- 问题超出 PRD 知识库范围（如竞品分析等）→ 兜底声明后提示"此类信息不在 PRD 知识库范围内"

**5. 缺陷主动提示**

回答问题的同时，如果发现：
- 问到的内容 PRD 中没有定义
- 存在明显的逻辑矛盾
- 某个边界条件没有覆盖

主动提示："⚠️ 发现潜在缺陷：[描述]，建议运行 `/update-prd` 补充。"

## 约束

- 本命令**只读**，不修改任何 PRD 或 context 文件
- 答案严格基于文档原文，不做额外推断、解释或延伸
- 草稿区来源必须 100% 标注 ⚠️ 警告，不得作为正式答案呈现
