## 触发条件（When）

用户提供历史PRD内容（任意格式）+ 对应需求ID，请求录入 prds/ 体系时。

触发方式：`/ingest-prd`，随后粘贴原始PRD内容并说明需求ID。

## 不适用场景（Do NOT use）

- 新需求（未上线）→ 改用 `/new-prd`
- 仅做需求澄清，不创建文件 → 改用 `/requirement-clarifier`
- 已有 prd.md 需要更新 → 改用 `/update-prd`

## 执行逻辑（Step by Step）

### Step 0：读取上下文

执行前必须读取：
- `context/iteration-requirement-list.md`：查找用户提供的需求ID对应的元数据（上线时间、优先级、标签、负责人）
- `prds/_registry.md`：确认当前最高 ID 编号，计算本次应分配的 ID
- `context/product-background.md`：理解产品线上下文，辅助推断缺失的产品线标签

### Step 1：确定 PRD 层级

根据以下规则自动判定，可被用户覆盖：

| 情况 | 层级 |
|------|------|
| 1条需求 + 功能简单（单页面/单场景） | story-card（SC-） |
| 2~6条需求 / 多场景 / 有明确页面结构 | feature-prd（F-） |
| 7条及以上需求 / 跨迭代主题 / 含多子功能 | epic-prd（E-） |

### Step 2：分配 ID 与生成文件路径

1. 从 `prds/_registry.md` 读取当前对应类型的最高编号，+1 得到新 ID
2. 文件夹命名规则：`[ID]-[中文功能名称]`，中文名从需求名称中提取核心词（≤10字）
3. 创建路径：`prds/[ID]-[中文名]/prd.md` 和 `prds/[ID]-[中文名]/CHANGELOG.md`

### Step 3：填写 frontmatter

```yaml
---
type: [story-card|feature-prd|epic-prd]
id: [SC/F/E-XXX]
title: [中文标题]
status: released
version: "1.0"
created: [最早关联需求的上线时间]
updated: [今天日期]
author: [负责人，从需求清单取]
feature-area: [从需求名称/标签推断]
product-line: [国内站|国际站|国内站,国际站|内部产品优化|待确认]
req-ids: [[需求ID1, 需求ID2, ...]]   # 关联的原始需求ID列表
epic: [所属 Epic ID，无则留空]
parent-feature: [所属 Feature ID，无则留空]
has-prototype: false
---
```

### Step 4：填写 PRD 正文

按对应模板结构填写，原则：

- **有内容**：直接写入，保留原意，优化表达使其符合模板规范
- **可推断**：结合需求名称、产品背景、领域模型推断，并在该段落后标注 `<!-- AI推断，请确认 -->`
- **确实缺失**：填写占位符 `> ⚠️ 待补充：[说明缺少什么内容，建议提供什么]`，**不杜撰**

Story Card 必填段：用户故事、验收标准
Feature PRD 必填段：用户故事、功能概述、页面&交互说明、业务规则、验收标准
Epic PRD 必填段：背景&业务目标、功能地图、分期规划

### Step 5：写入 CHANGELOG.md

```markdown
# Changelog

## v1.0 — [上线时间]

- 初始录入，来源：ePaaS签署迭代需求_v3.xlsx
- 关联需求ID：[需求ID列表]
- 录入日期：[今天]
```

### Step 6：更新 prds/_registry.md

在「活跃 PRD」表格中追加一行：

```
| [ID] | [标题] | [类型] | released | prds/[文件夹]/prd.md | [今天日期] |
```

### Step 7：更新 iteration-requirement-list.md

找到所有关联需求ID对应的行，将「关联PRD」列填入：`[ID](../../prds/[文件夹]/prd.md)`

### Step 8：输出录入报告

输出以下结构，不超过此格式：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📁 已创建：[ID] [中文标题]
   类型：[层级] | 路径：prds/[文件夹]/prd.md
   关联需求：[需求ID列表]
   产品线：[标签]（[来源：用户提供/AI推断]）

✅ 已填充：[列出已填充的章节]
⚠️  待补全：[列出标记了"待补充"的章节]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
❓ 缺口问题（共 N 条，请逐一确认）

Q1 [类别] 问题描述
Q2 [类别] 问题描述
...（最多5条，优先问影响AC和业务规则的）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 产品线推断规则

当需求在清单中标注为「未标注」时，依据以下关键词推断：

| 关键词 | 推断产品线 |
|--------|-----------|
| 国际站、eSignGlobal、香港、VIDA、VNPT、iamsmart、Adobe、多语言 | 国际站 |
| 国内站、支付宝、微信、钉钉、C2B、C2C、融合项目 | 国内站 |
| 灰度、埋点、稳定性、兼容性测试、性能优化、审计日志 | 内部产品优化 |
| 同时含国内+国际关键词 | 国内站,国际站 |

推断后在 Q1 位置列出，格式：
`Q1 [产品线] 根据关键词「xxx」推断为「xxx」，请确认或更正`

---

## 用户回答缺口问题后的处理

用户逐一回答 Q1~QN 后：
1. 将答案合并写入 prd.md 对应章节，移除 `⚠️ 待补充` 标记
2. 如答案引出新问题，最多追加 2 个新问题
3. 全部问题确认后，输出：`✅ [ID] 录入完成，可提供下一条`
