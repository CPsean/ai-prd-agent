## 触发条件（When）

用户提供历史PRD内容（任意格式）并请求录入 prds/ 体系时。

触发方式：`/ingest-prd`，随后提供原始PRD内容（需求ID可选，未提供时由 AI 从清单中匹配）。

## 不适用场景

- 新需求（未上线）→ 改用 `/new-prd`
- 仅做需求澄清，不创建文件 → 改用 `/requirement-clarifier`
- 已有 prd.md 需要更新 → 改用 `/update-prd`

---

## 执行逻辑（Step by Step）

### Step 0：读取上下文文件

执行前必须实际读取以下文件（不得使用记忆中的数据）：

1. `prds/_registry.md` — 确认当前最高 ID 编号，避免 ID 冲突（**必读**，失败则停止执行）
2. `context/iteration-requirement-list.md` — 匹配需求ID，获取上线时间/优先级/负责人（**可选**，见下方说明）
3. `context/product-background.md` — 辅助推断产品线（**可选**，缺失时跳过）

**关于 `iteration-requirement-list.md` 缺失的处理**：
- 文件存在 → 正常读取，用于 Step 0.5 的需求ID匹配
- 文件不存在或为空模板（只有表头/示例行）→ 输出以下提示后继续执行，所有需求按「情况C」处理：

> ⚠️ 未找到需求列表（`context/iteration-requirement-list.md`）。
> 建议先按 README Step 1.5 补充历史需求列表，可提升 ID 匹配准确率。
> 本次将跳过 ID 自动匹配，所有需求 ID 需手动确认。

> **路径注意**：工作目录为项目根目录，上述路径均为相对路径。

---

### Step 0.5：需求ID确认（正式执行前的前置确认）

在开始创建任何文件之前，先完成需求ID确认：

**情况A — 用户已提供需求ID**：在清单中查找，确认标题是否与PRD内容匹配，输出：
> `确认：需求 [ID]「[标题]」与本PRD内容匹配，继续执行。`

若不匹配，停下来告知用户，请用户确认正确的ID。

**情况B — 用户未提供需求ID**：从清单中按关键词搜索，列出最多3个候选：
> `在需求清单中找到以下候选，请确认哪个与本PRD对应，或告知"均不匹配"：`
> `· [ID1]「[标题]」— [所属迭代]`
> `· [ID2]「[标题]」— [所属迭代]`

**情况C — 清单中完全未找到匹配项**：标注为「未匹配」，继续执行，req-ids 置空，在 Step 8 报告中标记。

**情况D — 标题截断或信息不完整**：不得提前猜测写入。标注「标题截断，待补全」，待用户补充完整标题后再确认。

**情况E — 多版本文档（源文件包含V1/V2/V3等多版本）**：
- 询问用户：哪个版本是当前生效版本？历史版本有无对应需求ID？
- 确认后方可继续，不得自行判断保留哪个版本

---

### Step 1：确定 PRD 层级

根据以下规则自动判定，可被用户覆盖：

| 情况 | 层级 |
|------|------|
| 1条需求 + 功能简单（单页面/单场景） | story-card（SC-） |
| 2~6条需求 / 多场景 / 有明确页面结构 / 完整功能模块 | feature-prd（F-） |
| 7条及以上需求 / 跨迭代主题 / 含多子功能 / 架构底座类 | epic-prd（E-） |

判断后将理由记录，在 Step 8 报告中展示。

---

### Step 2：分配 ID 与生成文件路径

1. 从 `prds/_registry.md` 读取当前对应类型的最高编号，+1 得到新 ID
2. 文件夹命名：`[ID]-[中文功能名称]`，中文名从需求名称中提取核心词（≤10字）
3. 创建路径：`prds/[ID]-[中文名]/prd.md` 和 `prds/[ID]-[中文名]/CHANGELOG.md`

**关于日期**：created 字段以**需求清单中的上线日期**为准，不以源文件所在文件夹的日期为准。若清单中未找到该需求，以源文件的最早修订日期为准，并在 notes 中注明。

---

### Step 3：填写 frontmatter

```yaml
---
type: [story-card|feature-prd|epic-prd]
id: [SC/F/E-XXX]
title: [中文标题]
status: [released|in-review]   # 已在需求清单中 → released；待排期/未找到需求ID → in-review
version: "1.0"
created: [需求清单中的上线日期]
updated: [今天日期]
author: [负责人，从需求清单取]
feature-area: [从需求名称/标签推断]
product-line: ""   # 从 context/product-background.md 读取产品线列表，留空表示待确认
req-ids: [[需求ID1, 需求ID2]]   # 未找到匹配时置为 []
prev-req-ids: []                # 历史版本对应的需求ID（多版本文档时填写）
epic: [所属 Epic ID，无则留空]
related-stories:
has-prototype: false
notes: ""                       # 特殊情况说明（多版本、未匹配原因、待补全等）
---
```

**status 判断规则**：
- 需求ID存在于已完成迭代列表中 → `released`
- 来自「待排期PRD」文件夹 → `in-review`
- req-ids 为空（未找到匹配）→ `in-review`，在 notes 中说明原因

---

### Step 4：填写 PRD 正文

按对应模板结构填写，原则：

- **有内容**：直接写入，保留原意，优化表达使其符合模板规范
- **可推断**：结合需求名称、产品背景、领域模型推断，在该段落后标注 `<!-- AI推断，请确认 -->`
- **确实缺失**：填写占位符 `> ⚠️ 待补充：[说明缺少什么，建议提供什么]`，**不杜撰**
- **多版本文档**：只录入最新版本内容，CHANGELOG 中写历史版本演进摘要

**多端差异说明**：涉及国内站/国际站行为不同时，必须拆分为独立段落或表格，明确标注差异，不得合并描述。

Story Card 必填段：用户故事、验收标准
Feature PRD 必填段：用户故事、功能概述、页面&交互说明、业务规则、验收标准
Epic PRD 必填段：背景&业务目标、功能地图、关键决策记录

---

### Step 5：写入 CHANGELOG.md

```markdown
## v1.0 — [上线日期或录入日期]

- 初始录入，来源：[源文件路径]
- 关联需求ID：[需求ID列表]，录入日期：[今天]
- [多版本文档时]：本版本为 VX.0，历史版本：V1对应需求[ID]（[简短说明]）
```

---

### Step 6：更新 prds/_registry.md

在「活跃 PRD」表格中追加一行，备注栏按以下规则填写：

| 情况 | 备注内容 |
|------|----------|
| 已找到需求ID、已上线 | `需求[ID]；[关键特性一句话]` |
| 待排期、有预期迭代 | `待排期，预计[迭代名]；需求ID待补` |
| 完全未找到需求ID | `需求ID未匹配，[原因简述]；上线后补充` |
| 多版本、旧版已上线 | `V[N]对应需求[ID]已上线；V[N+1]待[迭代]` |

---

### Step 7：更新 iteration-requirement-list.md

找到所有关联需求ID对应的行，将「关联PRD」列填入：`[ID](../prds/[文件夹]/prd.md)`

若 req-ids 为空，跳过此步骤，在 Step 8 报告中说明。

---

### Step 8：输出录入报告

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📁 已创建：[ID] [中文标题]
   类型：[层级]（判断依据：[一句话说明理由]）
   路径：prds/[文件夹]/prd.md
   状态：[released|in-review]
   需求ID：[ID列表] | 来源：[用户提供|从清单匹配|未找到]
   产品线：[标签]（来源：[用户提供|AI推断]）

✅ 已填充：[已填充的章节列表]
⚠️  待补全：[标记了"待补充"的章节]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 产品线推断规则

> **产品线由用户在 `context/product-background.md` 中定义和维护，不在此文件中硬编码。**

执行时：

1. 读取 `context/product-background.md`，找到产品线定义及其关键词/标签
2. 根据 PRD 内容与产品线描述进行匹配推断
3. 若 `context/product-background.md` 尚未填写产品线信息，`product-line` 置空，在缺口问题中提示用户先完善上下文

推断后在缺口问题 Q1 列出确认，格式：
`Q1 [产品线] 根据关键词「xxx」推断为「xxx」，请确认或更正`

若上下文文件未初始化，格式：
`Q1 [产品线] context/product-background.md 中暂无产品线定义，请补充后重新执行，或手动指定本PRD的产品线`

---

## 用户回答缺口问题后的处理

用户逐一回答 Q1~QN 后：
1. 将答案合并写入 prd.md 对应章节，移除 `⚠️ 待补充` 标记
2. 如答案引出新问题，最多追加 2 个新问题
3. 全部问题确认后，输出：`✅ [ID] 录入完成，可继续下一条`

---

## 输出语言规范

- 所有输出、问题、说明**全部使用中文**
- 禁止使用技术英文缩写代替中文表达，例如：
  - ❌ i18n → ✅ 多语言支持
  - ❌ req-ids → ✅ 需求ID（正文描述时）
